## Context

当前普通用户账号模型仅保存手机号，登录方式是手机号加验证码；企业认证申请则绑定在已登录账号之后提交。目标流程要求把账号注册和企业资料提交合并成一次匿名注册申请，并以后台审核结果作为账号启用开关。这个变化横跨数据库模型、普通用户认证接口、小程序登录/注册流转，以及后台审核字段展示，因此需要在实现前统一约束。

## Goals / Non-Goals

**Goals:**
- 让匿名用户一次性提交登录名、密码、手机号和企业资料，形成待审核账号。
- 让普通用户登录仅在审核通过后成功签发 token；待审核或驳回账号不得进入业务首页。
- 保留后台审核中心，但补齐审核所需字段与账号状态回写。
- 为待审核和驳回账号提供无 token 的审核结果查询路径。

**Non-Goals:**
- 不引入短信验证码登录与找回密码流程。
- 不实现营业执照 OCR、实名核验或第三方企业信息校验。
- 不迁移历史手机号验证码账号到全新的企业账号体系之外的复杂兼容逻辑。

## Decisions

### 1. 将普通用户账号升级为“账号主体 + 审核状态”

`users` 表从仅有 `mobile` 的轻量记录升级为真正的账号表，新增 `username`、`password_hash`、`account_status` 等字段，并保持 `enterprise_applications` 作为审核记录表。

选择这个模型而不是把全部字段堆进 `enterprise_applications`，是为了清楚区分：
- 账号身份信息：登录名、密码摘要、手机号、账号门禁状态
- 审核申请信息：营业执照代码、法人信息、经营范围、经营地址、审核结论

替代方案是完全只存申请表，审核通过后再生成 users。该方案会让驳回重提、唯一性校验、密码更新和后台关联更复杂，因此不采用。

### 2. 登录接口拒绝未通过审核账号，不签发 token

`POST /api/auth/login` 改为接收 `username` 与 `password`。只有 `account_status=approved` 的账号才返回 token；`pending` 与 `rejected` 返回明确业务状态码和当前审核状态，不创建会话。

这样可以严格满足“未审核用户不能登录”的业务约束，同时避免出现“可登录但只能看审核页”的弱门禁模型。

替代方案是允许待审核账号登录后仅访问审核页。该方案虽然兼容现有前端更多，但与当前明确约束冲突，因此不采用。

### 3. 审核状态查询改为无 token 的注册申请查询

因为待审核账号无法登录，`/api/auth/audit-status` 不能继续依赖 Bearer token。接口改为使用注册后保存的 `applicationId`，并配合 `username` 或 `mobile` 做辅助校验，返回当前申请状态、驳回原因与最近提交资料摘要。

这样注册成功后可直接进入审核状态页，后续重新打开页面时也能基于本地缓存的申请标识查询，而不需要临时放宽登录限制。

替代方案是新增独立的“查询申请状态”接口并保留原 `audit-status` 鉴权接口。该方案会产生两套高度重叠的状态契约，因此本次直接重定义现有能力。

### 4. 驳回后重提沿用同一账号主体，生成最新审核申请视图

被驳回账号允许在原账号上修改资料并重新提交，最新一次提交覆盖账号当前可见的申请视图；后台审核列表和审核状态页都以“最近一次申请”为准。

这样可以保留登录名唯一性和账号主体稳定性，避免用户每次驳回都重新注册一个新账号。

### 5. 后台审核通过时同时回写账号门禁状态

管理员通过审核时，不仅更新申请状态为 `approved`，还同步将 `users.account_status` 更新为 `approved`；驳回时将其置为 `rejected`。注册时初始值为 `pending`。

这样登录准入只需要查询用户表，不必把申请表联表判断作为登录主路径。

## Risks / Trade-offs

- [历史登录方式破坏性变更] → 前后端规范中明确标记为 BREAKING，并同步修改小程序登录/注册入口文案。
- [待审核状态查询失去 token 保护] → 使用 `applicationId` 加注册标识双因子匹配，避免仅凭用户名即可枚举状态。
- [SQLite 迁移复杂度上升] → 采用幂等迁移策略，为既有 `users` 表补列而不是重建大表，并在设计中预留默认值。
- [驳回重提覆盖旧资料导致审计对比困难] → 保留 `enterprise_applications` 历史记录，并以最近一次记录作为当前有效视图。
- [密码安全要求提高] → 使用密码摘要存储，严禁明文存储或回传。

## Migration Plan

1. 为 `users` 表补充 `username`、`password_hash`、`account_status`、`updated_at` 等字段，并建立唯一索引。
2. 扩展 `enterprise_applications` 表，新增法人姓名、法人电话、经营范围、经营场所地址等审核字段。
3. 重写普通用户注册、登录、审核状态接口契约，并同步更新小程序页面与服务层。
4. 扩展后台审核详情与通过/驳回回写逻辑。
5. 用新流程验证：注册提交、待审核查询、审核通过登录、驳回重提。

## Open Questions

- 登录名是否允许使用手机号作为登录名，还是必须单独唯一命名。
- 审核状态查询接口最终采用 `applicationId + mobile` 还是 `applicationId + username` 作为辅助校验字段。
- 历史仅手机号登录的存量用户是直接停用，还是后续另开迁移变更补齐账号资料。
