## Context

当前项目已经形成了清晰的业务三层：`crawler` 负责采集，`server` 提供业务与管理接口，`miniapp` 承担正式用户端。管理能力虽然已经在 `server` 中部分存在，例如企业审核、企业目录、采集控制与运行记录接口，但仓库内缺乏一个正式定义、边界清晰的运营后台前端。

此前内部后台能力曾借助旧样例前端承载，但该样例工程已经退役。现在的问题不再是“后台要不要存在”，而是“如何从零定义一个正式的统一运营后台工程，并明确第一阶段必须具备哪些模块和页面”。这类规划如果不先做，会导致后续页面、导航、接口补齐和角色认知都持续漂移。

## Goals / Non-Goals

**Goals:**
- 定义统一运营后台第一阶段的正式信息架构
- 明确后台前端的核心模块、一级导航和关键页面
- 让企业审核、企业目录、采集控制、运行记录和总览形成统一的后台工作流
- 允许部分页面先以占位态或只读态落地，为尚未完备的 API 留出可扩展空间
- 为后续在仓库中新增 `admin-frontend/` 工程提供清晰的产品与技术边界

**Non-Goals:**
- 不在本 change 中直接实现后台工程
- 不要求本次同时补齐所有缺失 API
- 不在第一阶段做复杂 RBAC、多组织体系或完整 CRM
- 不把后台扩展成运维平台、BI 平台或客服平台
- 不改动 `miniapp` 的用户端职责边界

## Decisions

### 1. 在当前仓库内新建明确命名的正式后台工程，而不是复活历史样例前端
- **Decision:** 统一运营后台的规划目标是一个新的正式前端工程，例如 `admin-frontend/`，而不是继续沿用已经退役的样例目录。
- **Why:** 后台需要清晰的工程语义、明确的产品边界和独立的信息架构。复用历史样例会把“原型/样例”和“正式系统”再次混在一起。
- **Alternative considered:** 复活旧 `ui`。放弃，因为旧目录语义已经被用户明确否定，继续使用只会重复制造边界混乱。
- **Alternative considered:** 独立新仓库。暂不优先，因为当前更需要与 `server`、`crawler` 保持紧密协作，单仓成本更低。

### 2. 后台第一阶段按“工作流”组织，而不是按技术来源组织
- **Decision:** 后台导航与模块划分优先围绕运营工作流展开，包括“总览”“企业审核”“企业目录”“采集控制”“运行记录”，而不是简单按接口来源或数据库表拆栏目。
- **Why:** 运营后台的价值在于帮助人完成工作，而不是暴露技术结构。按工作流组织能让审核人员、运营人员和技术运营更快定位目标。
- **Alternative considered:** 按“用户管理/爬虫管理/API 管理”之类技术域分区。放弃，因为它对开发者友好，但对运营使用者不自然。

### 3. 把“可落地页面”与“API 完整度”解耦
- **Decision:** 页面规划必须先于 API 完整度；某些后台页面可以在第一阶段允许显示占位态、空态、等待补齐字段或仅读态，而不是等所有接口齐全后再定义页面。
- **Why:** 统一后台的难点之一就是服务端能力并不完全对称。如果把页面规划完全依赖 API 完整度，后台会永远停留在零散工具状态。
- **Alternative considered:** 只有接口已经存在的页面才能进入规划。放弃，因为这会遗漏大量真正必要的后台能力。

### 4. 后台基础壳层必须先成为正式 capability
- **Decision:** 后台基础壳层单独定义为 capability，包括登录入口、全局布局、环境标识、导航、面包屑、权限守卫、全局错误与空状态。
- **Why:** 如果没有统一壳层，后续每个业务模块都会各自长出自己的页面结构和访问控制，最终形成拼贴式后台。
- **Alternative considered:** 先做业务页面，再事后抽壳层。放弃，因为后台从第一天起就需要统一入口和统一工作方式。

### 5. 第一阶段后台必须覆盖四类核心模块
- **Decision:** 核心模块固定为：
  1. 后台总览
  2. 企业审核与企业目录
  3. 采集控制
  4. 运行记录与运行详情
- **Why:** 这四类模块直接覆盖了当前项目最核心的运营闭环：审核、企业状态、采集动作、运行状态。
- **Alternative considered:** 从企业审核单模块起步。放弃，因为这样会继续维持“局部工具”，而不是“统一运营后台”。

## Risks / Trade-offs

- [Risk] 一开始规划过多页面，导致后台范围膨胀  
  Mitigation: 明确只规划第一阶段核心模块，把 CRM、报表中心、权限中心等放到后续阶段

- [Risk] 规划出的页面需要尚未存在的 API，后续实现受阻  
  Mitigation: 在 spec 中允许“占位态 / 等待 API / 只读态”，同时把 API 缺口显式纳入后续任务

- [Risk] 统一后台同时服务运营和技术运营，角色目标不一致  
  Mitigation: 通过总览、审核、采集控制、运行记录等模块分区，而不是强行做一套完全统一的单视图

- [Risk] 当前管理员鉴权仍较轻量，正式后台可能暴露更多安全问题  
  Mitigation: 在第一阶段把权限入口与环境标识纳入壳层要求，并把更强的管理员认证体系列为后续扩展能力

## Migration Plan

1. 先完成统一运营后台的产品与能力定义，形成正式 OpenSpec。
2. 确定后台前端工程命名、目录位置和与 `server` 的对接方式。
3. 先实现后台壳层和一级导航，再按模块逐步落地页面。
4. 以现有 `/api/admin/*` 为第一批接入能力，同时记录缺失 API。
5. 后续分 change 或分迭代补齐接口、实现页面和扩展权限体系。

回滚策略：
- 这是规划性 change，不直接改动线上行为；
- 若后续后台实现方向变化，可以基于本 change 调整 capability 或单独提出新的后台规划 change。

## Open Questions

- 后台第一阶段是否需要一个正式的管理员登录页，还是继续沿用最小内部鉴权模型？
- 总览页第一阶段需要做到“可操作”，还是先作为信息汇总页即可？
- 企业目录未来是否会演进成更完整的企业运营台账，如果会，第一阶段是否要提前预留详情页结构？
- 采集运行记录是否需要与对账结果、告警状态进一步合并，形成统一的“运行健康中心”？
