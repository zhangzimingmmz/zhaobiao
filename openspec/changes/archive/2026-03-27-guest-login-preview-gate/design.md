## Context

当前小程序的页面链路已经包含启动页、登录页、首页和若干频道页，但访问控制仍然沿用“未登录可浏览、登录后再解锁操作”的模式。用户现在希望改为：

```text
launch
  ↓
未登录 → login
已登录 → home
```

其中“home / channel / detail / favorites”都属于完整数据页面，只有登录后才能进入；未登录用户只能停留在登录相关页面。不过，为了让小程序在上线前更容易通过审核，登录页不希望是一堵纯登录墙，而是要展示少量真实数据预览，让审核者和用户看到产品确实有内容。

因此，这个 change 的核心不是简单“把首页改成登录墙”，而是定义一种受控预览模型：

- 未登录用户可以看到 5 条真实数据轮播
- 但不能把预览区当成完整数据入口
- 登录成功后才进入首页与后续数据流

## Goals / Non-Goals

**Goals:**

- 让未登录用户在启动页后优先进入登录页，而不是首页。
- 在登录页中展示真实数据轮播预览，以提升审核可见性和产品感知。
- 保持首页、频道页、详情页和收藏页为受保护数据页面。
- 明确未登录预览与完整数据访问之间的边界，避免轮播区演变成匿名首页。

**Non-Goals:**

- 不修改后端登录/审核状态的业务语义。
- 不开放未登录详情页、频道页或完整筛选能力。
- 不在本次 change 中重做首页、频道页的视觉设计，只关注路由门禁与登录页预览整合。
- 不为未登录预览新增独立详情页或独立 API 资源模型，只在现有真实数据接口上做受控消费。

## Decisions

### 1. 加载页只负责首屏分流，不直接进入首页

**选择**：加载页结束后必须先判断 token：

- 有 token → 进入首页
- 无 token → 进入登录页

**理由**：

- 这是“登录后才能访问数据页面”的最上游入口。
- 如果加载页继续直接去首页，后续所有门禁都只能是补丁式兜底。

**替代方案**：继续先进入首页，再由首页判断是否跳登录。

- 被否决原因：会产生首页闪现、逻辑重复和体验不稳定。

### 2. 登录页预览区定义为 teaser，不定义为匿名数据入口

**选择**：登录页可以展示真实数据轮播，但该轮播区仅是受控预览，不承担完整数据浏览职责。

**规则**：

- 固定展示 5 条
- 使用真实列表数据
- 点击后仅提示登录查看详情
- 不允许通过轮播区进入详情页或频道页

**理由**：

- 这样既满足审核场景下“页面有内容”的需求，又不会破坏登录门禁。

**替代方案**：未登录时直接开放首页前 5 条数据并允许进入详情。

- 被否决原因：这会把预览区变成事实上的匿名数据入口，与登录门禁目标冲突。

### 3. 预览字段要轻，不要接近完整列表

**选择**：登录页预览卡片只展示：

- 标题
- 发布时间
- 来源
- 分类标签

**理由**：

- 这组字段足够让用户和审核者理解产品内容。
- 不会让登录页看起来像“完整首页列表缩略版”。

**替代方案**：展示预算、地区、截止时间、摘要等更多字段。

- 被否决原因：会使登录页预览过重，弱化登录动作本身，也更像未登录开放数据。

### 4. 使用真实数据，但要加轻量筛选约束

**选择**：预览区使用真实数据，但应只选择可稳定展示的最新 5 条记录，而不是裸取数据库任意前 5 条。

**建议约束**：

- 按发布时间倒序
- 标题非空
- 来源字段存在时优先展示

**理由**：

- 仍然符合“真实数据”要求
- 但能避免抽到异常数据、空标题或不可展示记录

### 5. 数据页面统一受保护，而不是仅拦收藏/详情动作

**选择**：以下页面统一视为受保护数据页面：

- 首页 `index`
- 工程建设 `construction`
- 政府采购 `government`
- 信息公开 `information`
- 收藏 `favorites`
- 招采详情 `detail`
- 信息详情 `info-detail`

未登录进入这些页面时，应统一被拦截到登录页或统一提示登录，而不是仅在某些页面的某些按钮上才校验 token。

**理由**：

- 当前代码的问题正是“只有局部动作受限，页面本身仍可访问”。
- 数据访问控制应是页面级规则，而不是按钮级规则。

### 6. 登录页仍然是表单优先，预览区是辅助区

**选择**：登录页的主角仍然是登录表单和注册入口，预览轮播位于表单下方，作为辅助区出现。

**理由**：

- 如果预览区压过表单，登录页会再次退化成“匿名首页”。
- 审核要看内容，但用户路径仍然应被引导到登录动作。

## Risks / Trade-offs

- [Risk] 登录页预览太重，会冲淡登录动作 → Mitigation: 预览只做 5 条、轻字段、不可进入详情。
- [Risk] 真实数据中可能出现不理想标题或不稳定内容 → Mitigation: 只取最新且可稳定展示的 5 条记录。
- [Risk] 页面级门禁如果分散实现，容易遗漏某些数据页 → Mitigation: 在设计中显式列出全部受保护页面，并统一处理。
- [Risk] 审核需要看内容，但用户又被限制访问 → Mitigation: 明确预览区就是审核和首次触达用的 teaser，而不是完整功能开放。

## Migration Plan

1. 先调整加载页后的分流逻辑，确保未登录优先进入登录页。
2. 再将首页、频道页、详情页和收藏页统一纳入页面级登录门禁。
3. 最后在登录页接入真实数据轮播预览，并验证点击行为仅提示登录。
4. 用未登录真机路径完整验证：launch → login → preview → login success → home。

## Open Questions

- 预览轮播是否混合展示全站最新 5 条，还是只展示某一频道最新 5 条。当前默认建议为“全站最新 5 条”。
