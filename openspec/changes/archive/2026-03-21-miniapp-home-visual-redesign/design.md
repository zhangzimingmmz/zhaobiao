## Context

当前小程序首页的数据流、筛选逻辑和一级导航已经基本稳定,问题主要集中在视觉层和信息架构层。首页上半屏连续叠加了一级分段、二级分段、招标计划/公告切换、搜索框和筛选按钮,其中关键控件直接使用 `taro-ui` 的 `AtSegmentedControl` 与 `AtSearchBar`,造成整体气质偏旧、偏后台、偏“条件查询面板”。

这次变更的目标不是重写业务规则,而是在保持现有分类、筛选与列表能力的前提下,重构首页的视觉语言和组件层级,让首页更像内容产品的主入口。

约束:

- 继续基于 Taro + 微信小程序,不切换技术栈。
- 继续允许 `taro-ui` 存在于表单/按钮等局部场景,但首页主视觉不再依赖其默认分段控件和搜索框。
- 本次不改后端 API,也不借 redesign 引入新的首页业务规则。

## Goals / Non-Goals

**Goals:**

- 以自定义组件替代首页最显老的默认控件骨架,建立更现代的频道导航与筛选语言。
- 为首页、收藏、我的等一级页建立更统一的 header / bottom-nav 视觉体系。
- 提升工程建设卡片与信息展示卡片的视觉层级,改善内容密度、留白和标签关系。
- 保留当前一级/二级分类、筛选能力和 API 绑定方式,尽量把变更集中在 UI 层。

**Non-Goals:**

- 不调整首页分类语义、筛选字段或接口契约。
- 不重做登录、注册、审核状态等表单页家族。
- 不在本次设计中引入动效库、主题切换或完整 design system 平台化。
- 不把当前个人主体小程序的“查看原文”能力问题与首页 redesign 混在一起解决。

## Decisions

### 1. 首页主控制区改为“频道 + 工具条”而非多层默认分段控件

首页顶部将从连续堆叠的 segmented controls 调整为更清晰的两层结构:

- 一级频道导航: 工程建设 / 政府采购 / 信息展示
- 当前频道下的次级切换与工具条: 次级切换、招标计划/公告胶囊、搜索和筛选

选择理由:

- 当前问题不在控件数量本身,而在于多层同构控件造成视觉疲劳。
- “频道 + 工具条”更符合内容产品入口的感知模型,也更容易区分层级。

备选方案:

- 继续保留 `AtSegmentedControl` 仅调颜色: 否决,因为问题是组件语言本身而非单纯配色。

### 2. 用自定义首页组件替换 `AtSegmentedControl` / `AtSearchBar` 的首页主视觉职责

首页中的一级导航、二级导航、招标计划/公告切换、搜索框将使用自定义组件,`taro-ui` 仅保留在表单类页面中。

选择理由:

- `taro-ui@3.3.0` 的 segmented/search 默认视觉较老,不适合作为首页品牌感来源。
- 自定义组件可以继续沿用现有数据流,只替换表现层,风险可控。

备选方案:

- 全局移除 `taro-ui`: 否决,因为本次 redesign 主要针对首页,没有必要扩大到整个项目。

### 3. 首页卡片重构为“标题区 + 关键信息 + 弱化元信息”三层模型

工程建设卡片与信息展示卡片都将保留现有字段支持范围,但重构层级:

- 第一层: 标题与收藏动作
- 第二层: 标签、来源、核心业务信息
- 第三层: 发布时间、截止时间等弱化元信息

选择理由:

- 现在卡片结构已经具备自定义基础,但留白和视觉重心不足。
- 不改字段支持范围也能显著提升观感,且不会影响 API 契约。

备选方案:

- 用大面积封面图或复杂瀑布流重做: 否决,因为与当前公告类内容密度不匹配。

### 4. 一级页面建立统一“轻品牌 header + 更明确 bottom-nav”语言

首页、收藏、我的等一级页将共享一套更有品牌感的 tab-page header 家族,同时 bottom-nav 调整为更现代的激活态和间距语言,但仍保持三个一级入口不变。

选择理由:

- 当前 top bar 与 bottom nav 虽已自定义,但还停留在偏工具化的基础状态。
- 首页大改若不连一级壳一起调整,容易出现“中间内容变新,外壳还像旧版”的割裂感。

备选方案:

- 只改首页列表区,不动 header / nav: 否决,因为无法真正改变整体第一印象。

### 5. 采用“token 增量重构”而非一次性全局换肤

视觉重构将优先扩展现有 `variables.scss`,调整颜色、间距、圆角、阴影和层级 token,再逐步替换首页相关组件。

选择理由:

- 现有小程序已经自定义了 TopBar、BottomNav、Card、Icon,具备 token 化改造基础。
- 增量重构比“一次性全局重做”更适合当前项目节奏。

备选方案:

- 全量引入新 UI 库或重新建一套组件平台: 否决,因为成本过高且与本次目标不匹配。

## Risks / Trade-offs

- [Risk] 首页视觉改动过大可能影响既有用户的操作习惯 → Mitigation: 保持一级/二级分类和筛选位置关系不变,重点调整表现层而非业务语义。
- [Risk] 自定义组件替换过多会引入样式回归 → Mitigation: 优先替换首页控制区和卡片,并做真机截图核对。
- [Risk] 颜色、阴影、圆角若控制不好会走向“花哨”而非“现代” → Mitigation: 保持内容导向,让视觉升级服务于层级而不是装饰。
- [Risk] redesign 可能诱发其他页面一起想重做 → Mitigation: 本 change 明确聚焦首页和一级页壳,表单页家族留到后续单独变更。

## Migration Plan

1. 先扩展首页相关视觉 token 和一级页壳样式基线。
2. 替换首页 primary / secondary / announcement-type / search 组件。
3. 重做首页卡片样式与列表区节奏,并保持 API 绑定不变。
4. 调整 tab-page header 与 bottom-nav 的统一视觉语言。
5. 通过设计验收截图和真机核对确认首页已明显摆脱旧式后台/表单感。

## Open Questions

- 首页是否需要在 redesign 后加入简短的品牌副标题或场景说明,还是维持更克制的标题区。
- 信息展示频道是否要在视觉上与公告频道进一步拉开差异,还是继续以统一壳为主。
