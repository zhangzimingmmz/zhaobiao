## Context

`miniapp-visual-parity-polish` 已经完成了第一轮视觉收敛，但最新模拟器截图显示页面仍然没有进入 PDF 对稿阶段。当前问题已经从“颜色、圆角、阴影不够像”演变为更底层的版式问题：

- 频道页存在重复头部链路，导致顶部被拆成多层，正文开始位置过晚。
- 首页与收藏页在长屏设备上首屏内容重心偏上，页面下半段出现大面积空白。
- 收藏页虽然是一级 tab，但还没有自己的版式结构，只是把通用 `SecondaryTabs` 和列表卡片拼接起来。
- 登录页仍然略高，表单与预览区的总高度在首屏内不够紧凑。
- 视觉验收流程还不够严格，当前甚至出现了“源码意图与截图结果不一致”的风险，说明需要把构建与截图核对纳入设计本身。

因此，这次 change 不再继续做“全局 token 微调”，而是把重心放到页面骨架与首屏节奏重排上。

## Goals / Non-Goals

**Goals:**

- 将频道页重排为单一头部、单一标题和紧凑列表的结构，消除重复层级。
- 给收藏页建立独立的一级列表页版式，使其与首页、频道页、登录页形成清晰家族差异。
- 为首页提供低权重支撑区块，以吸收长屏空白并稳定首屏重心。
- 将登录页进一步压缩到更接近 PDF 的首屏密度。
- 将截图与构建一致性纳入视觉验收规则，避免对着旧样式继续微调。

**Non-Goals:**

- 不改现有登录门禁、数据接口、筛选逻辑、收藏逻辑或详情数据契约。
- 不引入新的实时数据模块、推荐算法或首页运营内容体系。
- 不重做整套视觉 token；本次以版式重排和页面关系为主，token 仅作局部配合。
- 不要求像素级复刻 PDF，而是要解决重复头部、过大留白、卡片堆叠和页面缺少独立结构的问题。

## Decisions

### 1. 频道页采用“TopBar 标题 + 工具区 + 列表”单脊柱结构

**选择**：工程建设、政府采购、信息公开三类频道页统一只保留 `TopBar` 中的频道标题，不再在正文中重复渲染大标题块或过渡白层。

**结构：**

```text
TopBar(返回 + 居中频道标题)
↓
筛选工具区
↓
列表卡片
```

**要求：**

- `TopBar` 是频道名唯一的标题来源。
- 页面正文的第一视觉块必须是筛选工具区，而不是标题重复层。
- 第一张列表卡与工具区之间只保留轻量过渡间距。

**理由**：

- 当前截图中的主要违和点来自顶部被切成“蓝栏/白层/正文标题/筛选区”四段。
- 相比继续调整 padding，直接删除一整层结构更有效。

**替代方案**：保留正文大标题，移除 `TopBar` 居中标题。

- 被否决原因：会削弱频道页在微信导航胶囊环境中的页面识别度，也会让返回头部显得过空。

### 2. 收藏页归入列表页家族，但保留一级 tab 身份

**选择**：收藏页不再使用“tab 标题 + 裸 `SecondaryTabs` + 大卡片列表”的拼接结构，而是采用一级 tab shell 下的专用列表页布局。

**结构：**

```text
TopBar(收藏)
↓
收藏页局部 section
  ├─ 轻量类型切换条
  └─ 收藏列表
```

**要求：**

- 类型切换条必须作为收藏页局部 section 的一部分，而不是直接悬在 header 下方。
- 空态、游客态和列表态都保持当前选中分类上下文可见。
- 收藏页与频道页共享高密度列表节奏，但不复用频道页的“搜索 + 筛选”工具区。

**理由**：

- 收藏页当前最大问题不是颜色，而是“没有自己的版式”。
- 将其纳入列表页家族能复用卡片密度规则，同时保留 tab 页身份。

**替代方案**：继续只调整 `SecondaryTabs` 尺寸和 card margin。

- 被否决原因：不能解决“缺少独立页面结构”的根因。

### 3. 首页引入低权重支撑带吸收长屏空白

**选择**：首页在三张频道入口卡下方新增一个低权重支撑带，用于承接平台说明或使用提示，并在长屏设备上平衡首屏重心。

**要求：**

- 支撑带必须是低权重、静态或准静态内容，不得把首页重新做成数据列表页。
- 支撑带视觉上弱于频道入口卡，不得夺走首屏主重心。
- 入口卡本身继续收紧高度和间距，支撑带只是补平衡，不是替代收缩。

**理由**：

- 当前首页只有三张入口卡，长屏设备天然会留下大面积底部空白。
- 仅靠继续压缩卡片无法根治空白问题，需要补一个受控的第二内容层。

**替代方案**：不新增任何区块，只让空白作为背景留白存在。

- 被否决原因：实际截图已经证明这种做法在长屏手机上会显得页面没排满。

### 4. 登录页采用“首屏优先压缩”策略

**选择**：登录页继续保留 `intro + form + preview` 结构，但目标改为让首屏尽可能完整容纳标题、表单和预览起始区域。

**要求：**

- 标题区与表单卡的距离继续缩短。
- 表单卡的 padding、输入框高度、按钮高度都再压缩一轮。
- 预览模块维持辅助级别，优先保证 form 在首屏中的主导地位。

**理由**：

- 登录页当前已经接近 PDF，但纵向节奏仍然偏长。
- 相比重新设计模块，直接围绕首屏密度优化更稳妥。

**替代方案**：删除预览模块。

- 被否决原因：不符合现有审核与游客预览策略。

### 5. 验收改为“构建一致性 + 对稿截图”两步

**选择**：本 change 的验收必须先确认截图来自最新构建，再进行对稿判断。

**要求：**

- 每次视觉验收前都重新编译并刷新微信开发者工具预览。
- 对稿截图至少覆盖：首页、频道页、收藏页、登录页。
- 验收清单必须显式排查：
  - 重复头部或重复标题
  - 长屏大面积空白
  - 收藏页无专用版式
  - 筛选区和列表之间间距过大

**理由**：

- 当前已经出现“源码想要白顶栏，截图却仍然是蓝顶栏”的现象。
- 如果不先确认构建一致性，后续视觉讨论会继续失真。

**替代方案**：延续上一轮只看截图、不检查构建来源。

- 被否决原因：容易继续围绕旧样式做错误判断。

## Risks / Trade-offs

- [Risk] 首页新增支撑带后可能偏离 PDF 原始截图 → Mitigation: 将支撑带控制为低权重说明区，不引入数据主内容或运营模块。
- [Risk] 删除频道页正文大标题后，页面可能显得过于简单 → Mitigation: 保留 `TopBar` 居中标题并通过工具区与列表起始节奏建立层级。
- [Risk] 收藏页专用版式会引入额外样式分支 → Mitigation: 只新增 section 壳层，继续复用现有卡片与 tab 组件。
- [Risk] 继续压缩登录页可能导致可点击区域过小 → Mitigation: 保持微信可用最小点击尺寸，只压缩多余留白和重复边界。

## Migration Plan

1. 先更新 OpenSpec specs，明确单一头部、收藏页专用版式和首页支撑带等要求。
2. 落地共享 page shell：频道页、收藏页和首页分别建立新的首屏骨架。
3. 再调整登录页与卡片密度，保证所有首屏页面的垂直节奏回收一致。
4. 最后执行“重新构建 -> 刷新预览 -> 截图对稿”流程，完成第二轮视觉验收。

## Open Questions

- 首页支撑带最终使用“平台说明”还是“使用提示”文案更贴近产品语境。
- 收藏页类型切换条是否继续沿用现有 `SecondaryTabs` 组件，还是单独做一个更轻的 favorites switch。
