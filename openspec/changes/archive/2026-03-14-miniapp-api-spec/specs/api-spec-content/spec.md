# api-spec-content 规格

## ADDED Requirements

### Requirement: 列表接口契约完整

文档 MUST 定义列表类接口的完整契约：Method、Path、请求参数（至少包含分页参数与业务类型/分类参数）；筛选参数与首页各菜单组合一致（工程建设-工程建设、工程建设-政府采购、政府采购-采购意向公开、政府采购-采购公告、信息展示等），参数名与取值类型明确；响应结构包含总条数与当前页列表数组；列表单条字段表包含 id、标题、发布时间、来源/地区等展示与跳转详情所需字段，类型与可空性注明；字段 SHALL 可在《原始数据接口文档》的 site1-list 或 site2 列表响应对应到落库或展示字段。

#### Scenario: 首页各菜单列表可依文档请求

- **WHEN** 前端按「工程建设-招标计划」「工程建设-招标公告」「政府采购-采购公告」「采购意向公开」「采购公告」「信息展示」等组合请求列表
- **THEN** 文档中存在对应参数说明（如 category/noticeType 等），能构造出合法请求

#### Scenario: 列表单条含详情跳转与展示必需字段

- **WHEN** 前端渲染列表项并跳转详情
- **THEN** 文档中列表单条至少包含 id（或等价唯一标识）、title、发布时间、来源/地区等，以及详情页 path 或原文链接所需字段（如 linkurl、originUrl）

### Requirement: 详情接口契约完整

文档 MUST 定义招投标详情与信息展示详情的接口契约：Method、Path（含 id 或 planId 等 path/query 参数）；响应结构为单条对象；字段表包含详情页展示所需内容：公告类型、标题、发布时间、项目名称、预算金额、地点、招标人、代理机构、报名/截止/开标时间、公告正文、原文链接；信息展示详情含标题、发布时间、摘要、正文、公众号原文链接等。字段类型与可空性注明；SHALL 能对应到《原始数据接口文档》中 site1 详情或 site2 详情的响应字段。

#### Scenario: 招投标详情页所需字段在文档中有定义

- **WHEN** 实现招投标详情页
- **THEN** 文档中详情响应包含标题、类型、时间、项目信息、重要时间、正文、原文链接等字段定义

#### Scenario: 信息展示详情与公众号原文可依文档实现

- **WHEN** 实现信息展示详情页与「公众号原文」按钮
- **THEN** 文档中对应详情响应包含正文与原文/外链 URL 字段

### Requirement: 登录与验证码接口契约

文档 MUST 定义获取验证码与登录接口：验证码接口的 Method、Path、响应（如图片 URL 或 base64 的约定）；登录接口的 Method、Path、请求体（手机号、验证码等）、成功与失败响应结构（如 token、用户简要信息或错误码）。若后端尚未实现，文档仍须写出契约并可标注「Mock 约定」。

#### Scenario: 登录页可依文档组装请求与处理响应

- **WHEN** 实现登录页的获取验证码与提交登录
- **THEN** 文档中能查到验证码与登录的请求方式与响应字段，便于前端实现与 Mock

### Requirement: 注册与审核状态接口契约

文档 MUST 定义企业注册与审核状态查询接口：注册接口的 Method、Path、请求体（企业名称、统一社会信用代码、联系人、联系电话、营业执照等）；审核状态接口的 Method、Path、请求参数（如用户标识）、响应结构（审核中/通过/未通过及驳回原因等）。字段与注册页、审核状态页表单项及展示一致。

#### Scenario: 注册页与审核状态页可依文档对接

- **WHEN** 实现注册提交与审核状态展示
- **THEN** 文档中有注册请求体字段与审核状态响应字段定义

### Requirement: 收藏相关接口契约

文档 MUST 定义收藏列表与收藏/取消收藏接口（若产品需要）：列表的请求参数与响应结构；收藏/取消收藏的 Method、Path、请求参数（如 targetId、类型）。若当前不提供收藏接口，文档须明确说明「收藏仅本地」或「待定」，避免前端与后端理解不一致。

#### Scenario: 详情页收藏与我的收藏可依文档实现或明确仅本地

- **WHEN** 实现详情页收藏按钮与「我的收藏」列表
- **THEN** 文档中要么给出收藏接口契约，要么明确收藏能力范围（如仅前端本地）
