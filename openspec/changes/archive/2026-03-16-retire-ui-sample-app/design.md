## Context

当前仓库中的 `ui/` 并不是一个边界清晰的正式系统。它一方面包含与 `miniapp` 高度重合的用户端 Web 页面，另一方面又承载了 `/admin/reviews`、`/admin/companies`、`/admin/crawl` 等管理页面入口。用户已经明确说明：`ui` 只是借助其他工具生成的一个“小程序长相样例”，并非要长期维护或作为正式后台使用；正式用户端以 `miniapp` 为准。

这意味着本次变更不是“重构 `ui`”，而是“退役 `ui`”。设计重点不在于如何保留功能，而在于如何安全删除样例工程、收敛系统边界，并避免文档、需求和后续实现继续把 `ui` 当成正式交付物。

## Goals / Non-Goals

**Goals:**
- 明确 `ui` 属于样例工程而非正式产品代码
- 从仓库中移除 `ui` 目录及其相关正式定位
- 更新项目文档与架构描述，使正式系统只包含 `crawler`、`server`、`miniapp`
- 明确 `ui` 中现有 `/admin/*` 页面不再被视为正式后台实现
- 保留服务端管理接口，不把删除样例 UI 等同于删除后端管理能力

**Non-Goals:**
- 不新建替代性的正式 Web 管理后台
- 不在本 change 中重做 `/admin/reviews`、`/admin/companies`、`/admin/crawl` 的新前端
- 不修改 `miniapp` 业务能力范围
- 不删除后端管理接口或采集控制能力本身
- 不处理用户个人备份或历史产物恢复流程

## Decisions

### 1. 将 `ui/` 视为整体退役对象，而不是只删除其中的用户端页面
- **Decision:** 整个 `ui/` 目录按样例工程处理，统一从仓库移除，而不是尝试只保留 `admin/` 页面。
- **Why:** 用户已经明确表示 `ui` 本身不是正式后台，而只是样例代码。如果保留其中一部分，会继续制造“这个项目到底是不是正式后台”的歧义。
- **Alternative considered:** 只删除 `Home/Login/Register/...` 等用户端页，保留 `admin/` 页。放弃，因为这等于默认 `ui` 仍然是后台正式工程，与用户意图冲突。

### 2. 服务端接口保留，前端样例移除
- **Decision:** 删除 `ui` 不等于删除 `server` 中的管理员审核和采集控制接口；接口作为服务能力保留。
- **Why:** 当前真正的系统边界是“服务端能力”和“样例前端”分离。退役样例工程时，应避免误删已有服务能力。
- **Alternative considered:** 连带删除 `/api/admin/*`。放弃，因为本次 change 的目标是收敛前端边界，不是移除后台业务能力。

### 3. 文档中不再把 `ui` 归类为正式管理后台
- **Decision:** 所有架构、需求和联调文档都需要把 `ui` 从正式系统清单中移除，或显式标注其已退役。
- **Why:** 如果只删代码不改文档，后续会继续有人按旧认知补回 `ui` 或围绕它提需求。
- **Alternative considered:** 仅删除代码，不更新文档。放弃，因为这会留下持续性的认知债务。

### 4. 把“未来是否重建正式后台”留给新的 change
- **Decision:** 本次 change 只做退役，不预埋新的后台目录或过渡壳子。若未来需要正式后台，应单独提案。
- **Why:** 当前最重要的是先把边界收紧。把“删除样例”和“新建后台”绑在一起，会扩大范围并模糊决策。
- **Alternative considered:** 在删除 `ui` 的同时新建 `admin/` 或 `console/` 目录占位。放弃，因为这会暗示已有方案，但实际上产品和技术边界都还没重新定义。

## Risks / Trade-offs

- [Risk] 删除 `ui` 后，团队会暂时失去现成的 Web 可视入口  
  Mitigation: 在 proposal、design 和文档中明确这是样例退役而非功能消失；保留服务端接口，未来若需要正式后台再单独立项

- [Risk] 某些文档仍引用 `/admin/*` 页面路径，导致认知不一致  
  Mitigation: 在实施任务中加入文档清点，重点更新架构文档、需求清单和本地交互说明

- [Risk] 未来有人误以为 `ui` 与 `miniapp` 是双端战略的一部分  
  Mitigation: 在退役 spec 中明确 `miniapp` 是唯一正式用户端，`ui` 不再作为正式前端存在

- [Risk] 直接删整个 `ui` 目录会让一部分历史参考素材丢失  
  Mitigation: 用户已确认自己有备份；仓库层面不再承担保存样例产物的职责

## Migration Plan

1. 在 OpenSpec 中声明 `ui` 样例工程退役及相关 requirement 变更。
2. 盘点 `ui` 的职责并区分哪些是样例代码、哪些是仍应保留的服务端能力。
3. 删除 `ui/` 目录及其构建依赖。
4. 更新文档，把正式系统边界收敛为 `crawler`、`server`、`miniapp`。
5. 验证仓库中不存在残留的 `ui` 引用、脚本依赖或错误的系统定位描述。

回滚策略：
- 若删除后发现仍有必须依赖 `ui` 的流程，可从用户自有备份或 Git 历史恢复；
- 服务端接口保留不动，因此回滚只涉及前端样例目录和文档描述。
