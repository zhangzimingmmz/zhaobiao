## Context

当前小程序的认证流程已经拆成登录、注册、审核状态三个页面，但它们来自不同阶段的视觉迭代：登录页已经形成“品牌区 + 主任务卡 + 弱辅助信息”的较完整样板，而注册页和审核状态页仍然依赖旧的 secondary-page intro + 普通卡片结构。与此同时，整个小程序的全局母体仍是以 [TopBar](/Users/zhangziming/opt/projects/zhaobiao/miniapp/src/components/TopBar/index.scss)、[base.scss](/Users/zhangziming/opt/projects/zhaobiao/miniapp/src/styles/base.scss)、[variables.scss](/Users/zhangziming/opt/projects/zhaobiao/miniapp/src/styles/variables.scss) 为核心的白底、浅灰蓝背景、轻边框和政务蓝强调体系，因此认证页既不能继续各自为政，也不能演变为一套独立视觉系统。

## Goals / Non-Goals

**Goals:**
- 以当前登录页为认证页面家族的视觉样板，统一登录、注册、审核状态页的品牌区、主卡片、表单控件、按钮和文字层级。
- 允许三页保持不同布局和内容密度，以匹配“登录 / 长表单提交 / 状态展示”三类不同任务。
- 确保认证页家族继续服从整个小程序的白蓝轻量基线，而不是单独长出另一套设计语言。

**Non-Goals:**
- 不修改登录、注册、审核查询、重提交流程或 API 契约。
- 不重构首页、频道页、收藏页等非认证页面。
- 不新增验证码、找回密码、上传流程或新的认证状态。

## Decisions

### 1. 登录页作为 Auth Family 样板，而非继续单页特例
- 登录页保留当前 `TopBar -> 品牌区 -> 主任务卡` 的主结构，作为认证家族的视觉参考页。
- 后续注册页与审核状态页不复制登录页布局，但必须继承其关键视觉语言：品牌区语法、白底轻卡、输入框、按钮和辅助信息弱化方式。
- 登录页本身在本 change 中只做必要收口，不再继续叠加新模块或更强装饰。

**Why this over alternatives**
- 比把登录页收回到旧 secondary-page 表单页更合理，因为登录页已是当前小程序最完整的一页。
- 比让注册页和审核页各自继续独立迭代更稳，避免认证流程内出现三套风格。

### 2. 抽取认证页共享壳层，但保留各页不同布局
- 认证页家族共享：背景、顶部栏、品牌区语法、主卡片圆角/边框/阴影、主按钮、输入控件、状态色规则。
- 登录页保持短表单；注册页使用同类壳层承载长表单并允许滚动；审核状态页使用同类壳层承载状态主卡。
- 品牌区按三档尺度处理：登录页最大、注册页中等、审核状态页最轻。

**Why this over alternatives**
- 比完全共用一个页面模板更适合当前任务差异，不会把审核状态页硬做成表单页。
- 比只统一 token 而不统一壳层更清晰，实施时能直接围绕共享样式组织代码。

### 3. 注册页从“旧表单页”升级为“认证家族长表单页”
- 注册页删除卡片外的独立 intro 作为主要视觉入口，将标题、短说明和重提交提示收进注册主卡或品牌区附近。
- 注册主卡复用登录页表单控件语言：白底细边框输入框、统一 textarea、同一套主按钮尺寸和次级文案灰度。
- `isResubmit` 提示保留，但不再单独漂浮成另一类信息盒子，而是成为主卡内的轻提示模块。

**Why this over alternatives**
- 比继续保留旧 intro + tip + form 三段式更统一，也能减少长表单页的碎片感。
- 比把注册页完全压成和登录页一样短的卡片更合理，因为注册页天然内容更长。

### 4. 审核状态页改为“状态主卡页”
- 审核状态页移除页首独立 intro 和英文 eyebrow，将当前状态标题、说明、步骤、资料、驳回原因统一收进主状态卡。
- `pending / approved / rejected / none` 仍然保留不同状态语义，但状态色仅用于局部强调块，不改变整页母体。
- 主按钮 / 次按钮继续沿用认证页家族按钮体系。

**Why this over alternatives**
- 比保持当前“说明页 + 多个内容盒子”的结构更像同一条认证流程的结果页。
- 比让审核页完全套用登录页表单布局更合理，因为其核心任务是状态告知与下一步引导。

### 5. 认证页家族必须服从全站 design tokens，而不是绕开它们
- 所有新样式优先复用 `variables.scss`、`base.scss`、`app.scss` 中已有颜色、字号、圆角和阴影定义。
- 登录页品牌区允许保留较强识别度，但不得再引入脱离全站的新主色、重阴影或特殊容器体系。
- 注册页和审核状态页向登录页靠拢时，以上述 token 为上限，而不是简单复制所有像素值。

**Why this over alternatives**
- 比直接把登录页当前样式复制到其他页更稳，能避免整个小程序出现“认证页是另一套产品”的感觉。

## Risks / Trade-offs

- [Risk] 登录页当前视觉张力较强，若原样复制到注册页和审核页，可能让认证页家族脱离首页与频道页。  
  → Mitigation: 将登录页视为样板上限，注册页和审核页使用同语法但更克制的品牌区尺度和容器强度。

- [Risk] 注册页长表单如果全部收进单一主卡，可能在小屏设备上显得过长。  
  → Mitigation: 保持滚动布局，只统一样式语言，不强求首屏完整展示所有字段。

- [Risk] 审核状态页收拢为状态主卡后，步骤、资料、驳回原因等信息层级可能不够清楚。  
  → Mitigation: 在卡内维持清晰分区，用局部状态块和标题梯度代替页级分裂结构。

- [Risk] 现有 `miniapp-login-ui` 基础 spec 与当前真实登录页方向不一致，若不一起修正会影响后续归档。  
  → Mitigation: 本 change 同步修改 `miniapp-login-ui` requirement，使其与账号密码登录样板页一致。
