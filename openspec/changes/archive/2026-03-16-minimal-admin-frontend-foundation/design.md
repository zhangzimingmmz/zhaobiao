## Context

项目已经完成了统一运营后台的较大范围规划，但后续讨论明确把第一阶段收敛成“极简后台”：只服务一个运营人员，只保留 8 个核心页面，只做固定账号密码登录，只做最简单的壳层和最短业务闭环。与此同时，现有后端已经具备企业审核、企业目录、采集控制和运行记录的大部分 `/api/admin/*` 接口，但管理员登录和总览聚合接口仍存在缺口。

当前最需要解决的问题不是“后台要不要做”，而是“如何避免继续沿用偏重的旧任务清单，改为一版真正能尽快落地的最小后台方案”。

## Goals / Non-Goals

**Goals:**
- 明确极简后台的正式工程边界：`miniapp -> 用户端`、`admin-frontend -> 运营后台`、`server -> 后端`、`crawler -> 采集`
- 锁定第一阶段只做 8 个页面，不再扩展到更复杂的后台模块
- 确定后台登录采用固定账号密码，由后端校验，前端本地长期持久化
- 明确哪些现有 `/api/admin/*` 可以直接复用，哪些接口必须补齐
- 让前端实现顺序围绕“先跑通最小闭环”而不是“先做完整平台”

**Non-Goals:**
- 不做复杂管理员账号体系、角色权限体系和退出登录机制
- 不做企业详情页、设置页、消息页、报表页等扩展页面
- 不要求第一阶段解决所有后台可视化增强能力
- 不把 Dashboard 扩展成完整运营分析中心
- 不修改 `miniapp` 和 `crawler` 的现有职责边界

## Decisions

### 1. 新建 `admin-frontend/`，不复用旧前端目录
- **Decision:** 极简后台仍然作为一个新的正式工程落在当前仓库中，例如 `admin-frontend/`。
- **Why:** 即使范围变轻，也不应再复用已经退役的旧样例前端目录。
- **Alternative considered:** 继续扩展旧样例目录。放弃，因为会再次混淆“正式系统”和“历史原型”。

### 2. 第一阶段只做 8 个页面
- **Decision:** 第一阶段锁定为登录页、总览页、企业审核列表页、企业审核详情页、企业目录页、采集控制台页、运行记录列表页、运行详情页。
- **Why:** 页面数量锁死后，后台范围才不会继续膨胀。
- **Alternative considered:** 连企业详情页、设置页、统计页一起规划。放弃，因为这会再次回到偏重方案。

### 3. 登录放后端校验，前端只保存结果
- **Decision:** 管理员账号密码由后端接口校验，前端只负责登录表单、成功后持久化 token，并在本地长期保留登录态。
- **Why:** 用户已经确认账号密码校验应放后端；这样也比前端硬编码更干净。
- **Alternative considered:** 前端本地直接硬编码账号密码。放弃，因为安全边界更差，也不利于后续替换。

### 4. Dashboard 先走最小数据方案
- **Decision:** 总览页第一阶段只显示待审核数量和最近一次采集/运行概况；如果后端没有独立聚合接口，可先用前端聚合现有接口。
- **Why:** 这是当前最短闭环所需的信息，且不必因为缺少聚合接口阻塞页面上线。
- **Alternative considered:** 必须先补完整 `GET /api/admin/dashboard` 再做总览页。放弃，因为这会延迟最小后台落地。

### 5. 后台壳层只保留基础容器能力
- **Decision:** 后台壳层只做基础布局、简单导航、页面标题区、基础状态反馈，不做环境标识、复杂面包屑、全局搜索等。
- **Why:** 当前后台只服务一个运营人员，复杂壳层不会带来第一阶段收益。
- **Alternative considered:** 按更完整的中后台壳层设计。放弃，因为与当前极简目标不一致。

### 6. 先补真正阻塞极简后台的后端接口
- **Decision:** 后端接口优先级收敛成两类：
  1. 现有接口直接复用：`/api/admin/reviews*`、`/api/admin/companies`、`/api/admin/crawl/*`
  2. 需要补齐：`POST /api/admin/login`，以及视实现策略决定是否新增 `GET /api/admin/dashboard`
- **Why:** 不应把所有后台增强型接口都放到第一阶段。
- **Alternative considered:** 大量提前扩展后台接口。放弃，因为这会让后端补口范围失控。

## Risks / Trade-offs

- [Risk] 极简后台后续又被不断加页，重新膨胀
  Mitigation: 先锁定 8 个页面，并在 tasks 中只围绕这 8 个页面展开

- [Risk] 登录接口未及时补齐，前端无法真正开始
  Mitigation: 将 `POST /api/admin/login` 提升为第一批必须完成的后端任务

- [Risk] Dashboard 没有独立接口导致前端聚合逻辑偏散
  Mitigation: 第一阶段允许前端聚合，但要在 tasks 中明确记录是否最终补独立聚合接口

- [Risk] 旧的 `unified-admin-frontend-core` 任务清单与当前 change 冲突
  Mitigation: 本 change 明确作为更轻量、更接近实施的收缩版执行依据

## Migration Plan

1. 基于本 change 固定极简后台范围与必补接口范围。
2. 在仓库中新增 `admin-frontend/` 工程并先完成登录页与基础壳层。
3. 并行补齐 `POST /api/admin/login`，并确认 Dashboard 走前端聚合还是后端聚合。
4. 依次实现企业审核、企业目录、采集控制、运行记录 4 个工作流页面。
5. 记录缺失字段和后续增强点，但不阻塞第一阶段最小闭环上线。

回滚策略：
- 这是一个收缩性 change，本身不改线上行为；
- 若后续决定恢复更大范围后台规划，可回退到更完整的后台 change 继续拆分能力。

## Open Questions

- `GET /api/admin/dashboard` 是否在第一阶段就补成独立接口，还是先允许前端聚合现有接口？
- `POST /api/admin/login` 返回的 token 是否直接复用 `ADMIN_TOKEN` 模式，还是引入一套轻量管理员 token 生成逻辑？
- `admin-frontend/` 的技术栈是否直接沿用当前 repo 已熟悉的 React/Vite 方向，还是另外确定脚手架？
