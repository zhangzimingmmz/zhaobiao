## Context

当前小程序已经验证出两种不同的外链打开路径:

- 公众号文章链接可通过微信原生能力打开。
- 非公众号原文链接若依赖 `WebView` 打开自有 H5 或代理页,在个人主体真机环境下会被微信直接拦截。

现有实现中,详情页、文章详情页和部分主 spec 仍保留“有 `originUrl` 就直接跳 `WebView`”的假设,这会误导后续开发继续在 `webview-proxy` 上投入排障成本。这个 change 的目标不是立即恢复非公众号原文能力,而是把当前能力边界和未来企业主体切换后的启用条件建模清楚。

## Goals / Non-Goals

**Goals:**

- 将“个人主体下不可依赖 `web-view` 打开非公众号原文”写入正式规格。
- 将公众号文章打开路径明确建模为微信原生能力,而不是普通 `WebView`。
- 为未来企业主体切换后重新启用 `web-view + 业务域名` 定义前置条件、校验顺序和回归范围。
- 让后续实现以“能力开关 / 路由分流”为中心,而不是默认所有原文都直接跳 `WebView`。

**Non-Goals:**

- 本 change 不承诺当前个人主体小程序内恢复非公众号原文可打开。
- 本 change 不引入新的第三方依赖或新的抓取方案。
- 本 change 不在本阶段决定最终选择“复制链接”“浏览器打开”还是“正文提取展示”作为长期默认体验。

## Decisions

### 1. 将当前问题定义为平台能力约束,而不是单点实现缺陷

将真机实测结果作为规格输入: 个人主体下,自有 H5 探针页与代理页均无法在小程序 `WebView` 中打开。这样后续开发不会再把主要精力放在 `api/webview-proxy` 页面内容或 URL 拼装细节上。

备选方案:

- 继续把问题定义为代理页实现问题: 被否决,因为自有 H5 探针已经证明更上游的 `WebView` 能力不成立。

### 2. 原文打开策略改为“按链接类型 + 平台能力”分流

规格层面将原文打开行为拆成两类:

- `mp.weixin.qq.com` 文章: 使用微信原生能力打开。
- 其他原文链接: 只有在企业主体且业务域名验证通过后,才允许走 `WebView` / 代理页方案。

备选方案:

- 继续维持“只要有 originUrl 就跳 WebView”: 被否决,因为与当前真机结果冲突。

### 3. 将企业主体切换建模为显式启用条件

未来重新启用非公众号原文 `WebView` 能力时,必须同时满足以下条件:

- 小程序主体已切换为企业主体。
- 自有域名已完成 HTTPS、备案和微信业务域名校验。
- 真机已验证自有 H5 探针页可打开。
- 真机已验证代理页和至少一个真实原文站点可打开。

备选方案:

- 在代码中直接保留 `WebView` 路径,等待未来自然生效: 被否决,因为这种做法不会形成清晰的迁移门槛和验收标准。

### 4. 保留 `webview-proxy` 但下调其当前定位

`api/webview-proxy` 仍然是企业主体切换后的候选实现基础,因此不在规格层面删除。但在当前阶段,它不再被定义为个人主体下的可用能力。

备选方案:

- 立即从系统中删除代理页和相关代码: 被否决,因为未来企业主体切换后仍可能复用。

## Risks / Trade-offs

- [Risk] 规格记录了限制,但产品没有同时设计兜底体验 → Mitigation: 后续实现任务中要求明确个人主体下的降级方案。
- [Risk] 未来企业主体切换后,微信规则或后台入口与当前认知不一致 → Mitigation: 将“H5 探针页真机验证”设为重新启用前的第一关。
- [Risk] 仅修正文档但不调整代码,会继续保留错误行为 → Mitigation: 任务中要求同步更新详情页策略和运维说明。
- [Risk] `miniapp-article-display` 现有主 spec 与现状偏差较大 → Mitigation: 在本 change 中同时修正公众号文章打开方式的正式表述。
