# api-spec-doc 规格

## ADDED Requirements

### Requirement: 文档目标与适用范围明确

《接口文档-前端与小程序》MUST 在文档开头说明：本文档面向小程序与 Web 前端，描述其调用的**自有后端 API** 的契约；不描述爬虫与数据源（网站一、网站二）的原始接口；数据来源与字段设计参考《原始数据接口文档》与页面需求（如 PAGES_OVERVIEW）。

#### Scenario: 读者能区分本文档与原始数据接口文档

- **WHEN** 开发或产品阅读该文档
- **THEN** 能明确本文档是「前端/小程序调用的 API」，与《原始数据接口文档》的「爬虫-数据源」契约区分开

### Requirement: 接口清单与页面映射

文档 MUST 包含：（1）**接口清单**：列出所有前端/小程序需调用的接口（列表、详情、登录、验证码、注册、审核状态、收藏列表、收藏/取消收藏、以及可选的基础数据如地区/采购方式等），并注明每接口的 Method 与 Path（或 path 模板）；（2）**页面-接口映射**：标明首页、招投标详情、信息展示详情、登录、注册、审核状态、我的等页面分别依赖哪些接口及主要参数来源。

#### Scenario: 按页面可查到所需接口

- **WHEN** 实现某页面（如首页或详情页）时查阅文档
- **THEN** 能通过清单或映射表找到该页面需要调用的接口及入参来源（如列表的 category、分页、筛选参数）

#### Scenario: 清单覆盖列表与详情与认证与用户

- **WHEN** 对照 PAGES_OVERVIEW 与 miniapp-data 需求
- **THEN** 清单中至少包含列表类、详情类、登录/验证码、注册、审核状态、收藏相关接口

### Requirement: 每类接口的文档要素

文档中每个接口的说明 MUST 包含：接口标识或名称、Method、Path（含 path 参数若适用）、请求参数（query/body）表（参数名、类型、必填、说明）、请求体示例（若为 POST/PUT）、响应结构说明（顶层键、分页信息若适用）、响应字段表（字段名、类型、可空、说明）。字段说明 SHALL 在可行时注明与《原始数据接口文档》中落库字段或数据源字段的对应关系。

#### Scenario: 实现或 Mock 时能写出正确请求与解析响应

- **WHEN** 前端或后端根据文档实现某接口的调用或实现
- **THEN** 仅凭文档即可写出正确的 URL、参数与请求体，并解析响应（含字段类型与含义）

#### Scenario: 列表与详情接口含完整字段表

- **WHEN** 查阅列表或详情接口
- **THEN** 响应中用于列表展示与详情展示的字段均有表格列出，类型与可空性明确

### Requirement: 文档产出路径固定

文档 MUST 以仓库内单一 Markdown 文件形式产出；路径在变更的 tasks 或交付说明中明确（如 `docs/接口文档-前端与小程序.md` 或仓库根下 `接口文档-前端与小程序.md`），以便 miniapp-from-ui 及前端项目引用同一文件。

#### Scenario: 引用方可通过路径找到文档

- **WHEN** miniapp-data 或 README 引用「接口文档」
- **THEN** 使用约定路径即可定位到本接口文档文件
