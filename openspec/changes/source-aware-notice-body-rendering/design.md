## Context

当前 notices 表中的正文 `content` 来自两个形态差异很大的来源。`site2_ccgp_sichuan` 的正文几乎全部为完整 HTML，普遍包含 `div/p/style/table` 和来源站点的内联样式；`site1_sc_ggzyjy` 的正文几乎不含有效 HTML 标签，更像从网页抽出的长文本流，常见招标计划和招标公告内容已经失去可靠段落边界。详情接口当前将存储内容直接映射给小程序，导致 `site2` 在小程序 `RichText` 中继承大量不合适的 Web 样式，而 `site1` 即使做了轻清洗，仍会因为缺少结构信息而展示成一大段文本。

## Goals / Non-Goals

**Goals:**
- 为 `site1` 与 `site2` 建立不同的正文渲染策略，而不是继续用统一方案硬吃所有正文。
- 保留数据库中的原始正文，避免为展示优化污染采集原始数据。
- 让详情接口直接返回“小程序友好的已渲染正文”，使客户端消费逻辑简单稳定。
- 为格式化失败、异常正文和缺失内容建立明确降级路径。

**Non-Goals:**
- 不在采集阶段新增复杂正文抽取，也不改动 notices 原始入库结构。
- 不尝试 1:1 复刻原网站浏览器排版效果。
- 不在客户端新增按站点解析正文的复杂逻辑。

## Decisions

### 1. 在读取链路进行正文格式化，而不是改写数据库原文
详情接口在返回 `content` 时执行正文格式化，原始 notices.content 继续保留。这样可以立即覆盖历史数据、支持后续规则迭代，并在格式化策略有误时安全回退。

备选方案：
- 入库时清洗并覆盖原文：会污染原始采集数据，不利于回退。
- 客户端实时按站点重排：逻辑分散在小程序端，调试和一致性都更差。

### 2. `site2` 采用“轻清洗 + 统一样式”的 HTML 策略
对 `site2` 正文保留 HTML 结构，移除脚本、来源样式和无关属性，只保留适合小程序 `RichText` 的标签与属性；统一补充段落、标题、列表、表格、图片的渲染样式。这样可以最大化复用来源正文的语义结构，尤其适合采购公告与采购意向中的 HTML 表格。

备选方案：
- 纯文本化重排：会丢失大量表格和结构信息，不适合 site2。

### 3. `site1` 采用“编号/字段驱动的结构化分段”策略
对 `site1` 正文不再假设存在可靠 HTML，而是把它作为文本源处理。格式化器按编号前缀（如 `1.1`、`2.1`、`第X章`）和字段关键词（如“项目名称”“项目概况”“联系方式”“资金来源”）切段，输出稳定段落块；对于高度模板化的招标计划类文本，优先保留字段边界与阅读顺序，而不是保留原站网页观感。

备选方案：
- 只改 CSS：无法解决文本流缺少段落边界的问题。
- 为 site1 强行构造 HTML 表格：实现复杂且对泛化帮助有限。

### 4. 客户端只消费渲染结果，不自行解析站点正文
小程序详情页继续使用统一正文容器，但不再假设 `content` 是原始 HTML，也不做站点特有的切段规则。客户端只负责空态与渲染兜底，保持逻辑简单。

### 5. 为异常正文定义统一降级规则
若 `site2` HTML 清洗后为空，则退回为纯文本段落输出；若 `site1` 文本结构化失败，则至少按句号、分号、空行和常见编号模式切成段落；若正文完全为空，则返回 `null` 让页面进入空态。

## Risks / Trade-offs

- [site1 分段规则误切或漏切] → 先覆盖高频模式（编号、章节、字段名），并通过样本回归测试持续补规则。
- [site2 清洗过度导致信息丢失] → 仅移除脚本、样式和无关属性，优先保留表格、列表、链接、图片等核心标签。
- [服务端格式化逻辑继续膨胀] → 将站点策略拆成独立 helper，避免在详情接口里堆叠条件分支。
- [旧数据与新规则长期耦合] → 保留原始内容不变，允许后续规则迭代和重放。

## Migration Plan

- 先在服务端引入按站点格式化器，并在详情接口中切换到已渲染正文输出。
- 使用历史样本对 `site1` 招标计划、招标公告以及 `site2` 公告、采购意向进行回归验证。
- 若线上出现误格式化，可快速回退到上一版读取逻辑，因为数据库原始正文未被改写。

## Open Questions

- `site1` 招标计划是否需要进一步演进为“字段化展示”而不只是结构化正文。
- `site2` 采购意向中的超宽表格是否需要额外转为卡片或键值对展示。
