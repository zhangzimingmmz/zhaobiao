# 生产部署架构

本文定义后端在生产-like 环境中的标准部署形态。当前推荐方案为 **`docker compose + scheduler`**，用于在单机上稳定运行 API、crawl dispatcher 和 routine crawler，同时保留后续迁移到其他服务器时的可移植性。公网暴露由 `100.64.0.7` 上独立部署的 Traefik 负责，应用节点仍为 `100.64.0.5`。

## 1. 适用范围

当前方案覆盖：

- FastAPI 后端容器化运行
- 内置 crawl dispatcher 的单实例运行约束
- `site1` / `site2` 例行 `incremental` 与 `recovery` 定时调度
- `admin-frontend` 运营平台静态部署
- SQLite 数据文件与日志目录持久化
- 新环境部署、迁移与回滚

当前方案不覆盖：

- 多副本高可用
- 外部数据库迁移
- 独立 worker 拆分
- 自动执行高风险维护动作（如 `site2 backfill --formal`）

## 2. 架构总览

```text
                     ┌───────────────────────────┐
                     │  Client / Tailscale 内网  │
                     │  :8000(api) / :8091(ui)   │
                     └───────┬─────────┬─────────┘
                             │         │
                             ▼         ▼
                    ┌───────────────────┐   ┌──────────────────────┐
                    │   api container   │   │ admin-frontend       │
                    │ FastAPI +         │   │ static nginx         │
                    │ in-process        │   │ Vite build output    │
                    │ dispatcher        │   └──────────────────────┘
                    └─────────────┬────────────────┘
                                  │
                     claim queued │ run
                                  ▼
                   python -m crawler.site1/site2.tasks.*

                    ┌──────────────────────────────┐
                    │      scheduler container     │
                    │  supercronic + submit script │
                    └─────────────┬────────────────┘
                                  │
                         POST /api/admin/crawl/runs
                                  │
                                  ▼
                         crawl_runs queue / history

                    ┌──────────────────────────────┐
                    │         persistent data      │
                    │  /app/data + /app/logs       │
                    └──────────────────────────────┘
```

## 3. 服务划分

### 3.1 `api`

职责：

- 对外提供 HTTP API
- 启动本地 dispatcher
- 认领并执行控制面中的 queued run
- 持有 SQLite 数据文件与 `logs/admin-crawl` 输出

关键约束：

- **只能单实例运行**
- **不能使用 uvicorn 多 worker**
- **必须使用整仓库镜像**，因为运行时会直接调用 `crawler.*`

### 3.2 `scheduler`

职责：

- 按计划触发 routine 任务
- 通过管理控制面提交 run request
- 不直接绕过控制面执行 `python -m crawler...`

这样做的好处：

- 保留统一 run history
- 复用现有 action 白名单与参数边界
- 复用站点级排他保护

### 3.3 `data` / `logs` 持久卷

用途：

- `data/notices.db`：SQLite 数据库
- `logs/admin-crawl/*.log`：控制面运行日志
- 其他 crawler 日志也可保留在同一挂载目录下

## 4. 代码与运行时前提

当前代码结构决定了部署时必须满足以下前提：

- `server.main` 在启动时创建 dispatcher
- `server.crawl_control` 通过子进程执行 crawler 任务
- 数据库存储仍为本地 SQLite
- `site2` 运行时依赖 `curl_cffi`
- 管理员认证默认值仅适用于开发，不可用于生产-like 环境

因此，部署镜像必须同时安装：

- `server/requirements.txt`
- `crawler/requirements.txt`

以及当前 crawler 运行所需的完整依赖。

## 6. 环境变量

先复制环境文件：

```bash
cp .env.backend.example .env.backend
```

至少需要替换：

- `ADMIN_TOKEN`
- `JWT_SECRET`

当前第一阶段中，scheduler 直接复用 `ADMIN_TOKEN` 调用控制面；本 change 不单独引入 scheduler 专用鉴权令牌。

关键变量说明：

- `API_PUBLISH_BIND`
  - 控制容器暴露地址
  - 目标机 `100.64.0.5` 上推荐设为 `100.64.0.5:8000`
- `PUBLIC_ADMIN_DOMAIN`
  - 运营平台公网域名
  - 推荐设为 `admin-zhaobiao.zhangziming.cn`
- `PUBLIC_API_DOMAIN`
  - 后端 API 公网域名
  - 推荐设为 `api-zhaobiao.zhangziming.cn`
- `API_BASE_URL`
  - scheduler 容器访问 API 使用
  - Compose 内默认 `http://api:8000`
- `ADMIN_FRONTEND_PUBLISH_BIND`
  - 运营平台前端暴露地址
  - 推荐设为 `100.64.0.5:8091`
- `ADMIN_FRONTEND_API_BASE`
  - 前端构建时写入的后端 API 基地址
  - 推荐设为 `https://api-zhaobiao.zhangziming.cn`
- `DATA_DIR`
  - SQLite 和其他数据持久目录
- `LOG_DIR`
  - crawler / control-plane 日志目录
- `ADMIN_CRAWL_BACKFILL_MAX_DAYS`
  - 控制面允许的 backfill 最大窗口
- `SCHEDULER_SUBMIT_TIMEOUT_SECONDS`
  - scheduler 调用 API 的超时时间

## 7. 启动方式

统一使用：

```bash
docker compose --env-file .env.backend -f docker-compose.backend.yml up -d --build
```

查看状态：

```bash
docker compose --env-file .env.backend -f docker-compose.backend.yml ps
docker compose --env-file .env.backend -f docker-compose.backend.yml logs -f api
docker compose --env-file .env.backend -f docker-compose.backend.yml logs -f scheduler
```

停止服务：

```bash
docker compose --env-file .env.backend -f docker-compose.backend.yml down
```

## 8. 例行调度策略

当前默认计划定义在 [deploy/scheduler/crontab](/Users/zhangziming/opt/projects/zhaobiao/deploy/scheduler/crontab)：

- `site1.incremental`：每 2 小时
- `site1.recovery`：每天 03:40
- `site2.incremental`：每 2 小时
- `site2.recovery`：每天 06:00

提交路径为：

```text
scheduler -> POST /api/admin/crawl/runs -> dispatcher -> crawler subprocess
```

这意味着：

- scheduler 不直接运行 CLI
- 所有 routine run 都进入统一 `crawl_runs`
- 人工触发和定时触发共享同一套站点级互斥保护

## 9. 手工维护任务边界

以下动作**不建议纳入自动调度**：

- `site2 backfill --formal`
- 大窗口 backfill
- `precheck`
- `cleanup`
- `reconcile`

这些继续通过容器内手工执行：

```bash
docker compose --env-file .env.backend -f docker-compose.backend.yml exec api \
  python -m crawler.site2.tasks.reconcile --start 2026-03-01 --end 2026-03-15
```

或：

```bash
docker compose --env-file .env.backend -f docker-compose.backend.yml exec api \
  python -m crawler.site2.tasks.backfill --start 2026-03-14 --end 2026-03-15
```

## 10. 健康检查与单实例约束

当前没有专用 `/health` 路由，因此 Compose 使用对 `/openapi.json` 的轻量探测作为 HTTP 健康检查。

重要约束：

- 不要给 `api` 服务做副本扩容
- 不要为 uvicorn 配置多 worker
- 不要让 scheduler 绕开控制面直接运行 crawler CLI

前端服务为静态站点，采用 `try_files ... /index.html` 的 SPA 回退策略，因此 `/dashboard`、`/reviews` 等前端路由可以直接访问。

## 11. 首次部署验证

建议按以下顺序检查：

1. `docker compose ... config` 成功渲染
2. `api` 容器启动成功并变为 healthy
3. `scheduler` 容器启动成功
4. 能访问：

```bash
curl http://100.64.0.5:8000/openapi.json
```

5. scheduler 或手工执行以下命令后，`crawl_runs` 中出现新的 run：

```bash
docker compose --env-file .env.backend -f docker-compose.backend.yml exec scheduler \
  python /app/scripts/scheduler_submit.py site2.incremental
```

6. 查看 API 日志和 `logs/admin-crawl`，确认 run 有对应日志落点

## 12. 迁移步骤

迁移到新服务器时：

1. 复制整仓库
2. 安装 Docker 与 Docker Compose
3. 复制 `.env.backend`
4. 准备 `data` 与 `logs` 持久目录
5. 执行 `docker compose --env-file .env.backend -f docker-compose.backend.yml up -d --build`
6. 用 `scheduler_submit.py` 或控制面确认 routine submit 正常

如果需要迁移旧数据：

- 直接迁移 `data/notices.db`
- 同时保留 `logs/` 目录用于排障和历史核对

## 13. 回滚策略

如果 Compose 部署失败：

1. 停掉容器：

```bash
docker compose --env-file .env.backend -f docker-compose.backend.yml down
```

2. 保留 `data` 与 `logs` 挂载目录不动
3. 回到原有单机源码运行方式
4. 使用同一份 `data/notices.db` 与 `logs/` 继续运行

由于当前数据库仍为 SQLite，且部署方案只改变运行载体，不改变数据模型，因此回滚不需要额外数据迁移。

## 14. 已知限制

- 当前仍是单机 SQLite 架构
- `api` 是单点
- scheduler 依赖 API 可用性
- 代理配置和外部网络质量仍然是 crawler 运行前提
- 若未来需要多副本和更严格调度隔离，应进入第二阶段：拆分独立 worker，并评估 SQLite 之外的队列/数据库方案
