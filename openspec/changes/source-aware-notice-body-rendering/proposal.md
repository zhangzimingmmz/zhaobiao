## Why

当前公告正文展示存在两类完全不同的数据形态：`site2_ccgp_sichuan` 提供完整但带大量 Web 端样式的 HTML，`site1_sc_ggzyjy` 则大多是已塌缩成文本流的正文内容。继续使用单一的“原样返回或只做轻度清洗”策略，会让 site2 保留过多噪音样式，也无法解决 site1 正文在小程序中连成一整段、可读性很差的问题。

## What Changes

- 为公告正文增加按来源站点区分的渲染策略：`site2` 走 HTML 轻清洗与统一样式，`site1` 走文本结构化分段与纯文本重排。
- 详情接口返回的小程序正文内容从“原始正文”改为“面向小程序渲染的正文结果”，但继续保留数据库中的原始正文不变。
- 招投标详情页与信息详情页明确以服务端渲染结果为准，不再在客户端尝试按站点解析正文结构。
- 为正文格式化失败、缺失或极端脏数据补充统一降级规则，确保页面至少返回稳定可读的段落内容。

## Capabilities

### New Capabilities
- `notice-body-rendering-strategy`: 定义公告正文在读取链路上的按来源渲染策略、降级规则与输出约束。

### Modified Capabilities
- `notices-api`: 详情接口中的 `content` 语义从原始正文调整为小程序友好的已渲染正文，同时保持原始存储不变。
- `miniapp-notice-detail-pages`: 详情页正文区改为直接消费服务端渲染结果，并约束客户端不再做站点特有的正文解析。

## Impact

- 受影响代码：`server/main.py` 详情映射与正文格式化逻辑、小程序详情页正文消费逻辑、相关接口测试。
- 受影响接口：`GET /api/detail/bid/:id`、`GET /api/detail/info/:id`。
- 受影响系统：公告详情展示链路（采集数据读取 → API → 小程序详情页）。
