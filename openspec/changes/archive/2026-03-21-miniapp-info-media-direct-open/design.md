## Context

当前小程序首页的信息展示列表已经接入文章接口,但产品语义仍然混杂着两套模型:

- UI 上,信息卡片有时像“站内详情页入口”,点击后进入 `info-detail` 中转页;
- 数据上,这些内容本质上主要是公众号文章,真正的完整内容仍在 `wechatArticleUrl` 对应的原文页面。

与此同时,信息卡片的媒体区缺少明确策略。文章可能存在三种现实状态:

- 有有效封面图;
- 没有封面图;
- 封面字段存在,但图片 URL 加载失败。

当前实现没有把这三种状态显式建模,因此会出现卡片高度不稳定、封面位残留灰色空窗、以及用户多走一层无实质价值的详情页的问题。

约束:

- 继续基于当前文章数据模型,不引入新的文章正文抓取或站内富文本沉淀。
- 继续兼容少量没有 `wechatArticleUrl` 的记录或特殊测试记录,不能把所有信息详情页直接删除。
- 继续保持小程序个人主体下的能力边界,不重新引入 `web-view` 作为公众号文章主打开方式。

## Goals / Non-Goals

**Goals:**

- 为信息展示卡片定义稳定的媒体态策略,显式支持有图、无图、坏图降级三种表现。
- 让信息卡片在混合媒体状态下仍保持较稳定的列表节奏,减少忽高忽低的观感。
- 将带 `wechatArticleUrl` 的信息卡片主点击行为改为直接打开公众号原文。
- 将信息详情页收缩为兜底页,只在缺少原文链接或特殊记录时继续承接。

**Non-Goals:**

- 不在本次变更中为公众号文章建立完整站内正文页。
- 不在本次变更中改变文章后台录入流程或引入图片代理服务。
- 不处理招投标详情页的原文打开策略,该问题仍由其他 change 单独跟进。

## Decisions

### 1. 信息卡片采用“统一骨架 + 可退化媒体区”而不是完全双轨卡片

信息卡片继续共享同一张卡片骨架,但媒体区从“有图才显示/无图则缺失”调整为可退化模块:

- 有图: 显示真实封面;
- 无图: 显示设计化的文章占位块;
- 坏图: 在前端图片加载失败后自动退化为无图占位块。

选择理由:

- 比完全分裂为两套卡片样式更容易保持首页视觉一致性。
- 比保留灰色空窗更符合“无图文章是正常内容状态”的语义。

备选方案:

- 直接把无图卡片做成另一套纯文本卡: 否决,因为会进一步放大列表高度差异与视觉跳变。

### 2. 信息卡片高度采用“半固定策略”而不是完全自由或完全等高

卡片中的标题、摘要、时间等文本区域将使用稳定的行数上限和最小高度,卡片整体使用统一的 `min-height`,但允许在少量极端内容下自然增长。

选择理由:

- 完全自由高度会继续放大有图/无图、长标题/短标题的差异。
- 完全等高会迫使无图内容保留大量空白,显得机械且浪费空间。

备选方案:

- 所有卡片完全固定高度: 否决,因为对长标题和摘要的截断成本过高。

### 3. 公众号文章卡片点击优先直接打开原文

当记录存在 `wechatArticleUrl` 且属于公众号文章时,卡片主点击行为将直接调用微信原生公众号文章打开能力,不再先进入 `info-detail`。

选择理由:

- 当前 `info-detail` 对公众号文章几乎不提供额外独立信息价值。
- 减少一次中转可以显著降低操作摩擦,更符合“分发入口”而非“站内沉淀页”的定位。

备选方案:

- 继续所有卡片先进详情页: 否决,因为详情页对公众号文章主要是重复展示标题、摘要和一个“查看原文”按钮。

### 4. 信息详情页仅作为兜底路径保留

`info-detail` 不删除,但职责收缩:

- 无公众号原文链接时可继续展示基础详情;
- 特殊测试记录或未来非公众号内容仍可复用该页;
- 对普通公众号文章,不再作为默认入口。

选择理由:

- 这样可以避免一次性删除现有页面和路由,同时保留例外内容的承接能力。

### 5. 封面字段空值与坏图在前端表现上统一为“无图态”

接口仍允许 `coverImageUrl` 为空,前端不把“空值”和“加载失败”区分给用户,统一收敛为无图态占位。

选择理由:

- 用户不需要理解“是真没图还是图坏了”。
- 前端列表可以把图片资源质量问题吸收掉,避免出现长期灰框或 broken image 感。

## Risks / Trade-offs

- [Risk] 直接打开公众号文章会减少站内详情页曝光 → Mitigation: 接受信息展示作为“原文入口”而非“站内沉淀页”的定位调整。
- [Risk] 无图占位块如果设计过重,会让无图文章看起来像假图片 → Mitigation: 使用轻量品牌占位,强调文章属性而非模拟真实封面。
- [Risk] 图片加载失败在弱网场景下会频繁发生 → Mitigation: 前端使用一次性失败降级,避免反复闪烁或重试噪音。
- [Risk] 少量无原文链接的旧数据仍需详情页 → Mitigation: 保留 `info-detail` 作为兜底页,并在卡片层按是否有链接决定行为。

## Migration Plan

1. 先更新信息卡片 requirement,明确媒体态和点击路径的新规则。
2. 在小程序卡片组件中加入无图占位与图片失败降级逻辑。
3. 将首页信息卡点击逻辑改为“有原文直开,无原文再进详情页”。
4. 调整信息详情页,使其围绕兜底场景而不是公众号文章主场景组织。
5. 真机验证有图、无图、坏图、无原文四类文章行为。

## Open Questions

- 无图占位块是否需要根据分类显示不同文案或图形,还是统一为“公众号文章”占位。
- 卡片是否还要保留摘要两行,还是在无图态下改成更强调标题的紧凑布局。
