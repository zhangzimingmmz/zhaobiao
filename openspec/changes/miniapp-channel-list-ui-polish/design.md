## Context

频道页在完成头部链路修正和基础密度收紧之后，已经不再出现“双导航”这种结构性问题，但列表仍然不够高效：标题偏长、每条卡片的结构几乎完全重复、辅助信息不够弱，导致用户很难快速扫一眼判断“这条值不值得点进去”。与此同时，顶部筛选模块虽然功能齐全，但主次关系仍不够明确，搜索框没有成为主视觉，tab 和筛选按钮更像一组并列控件。

这次变更只聚焦频道页的视觉扫读效率和 UI 层级，不新增新的筛选维度、搜索联想、排序能力或业务状态字段。实现范围集中在频道页共享工具区和 `BidCard` 的视觉组织方式。

## Goals / Non-Goals

**Goals:**
- 提升频道页的扫读效率，让用户能更快通过标题、类型标签、来源和日期判断列表价值。
- 让筛选模块成为更明确的主工具区，建立“搜索框最重要，tab 与筛选按钮次之”的层级。
- 进一步收紧 `BidCard` 的重复感和信息厚度，让列表更像高效公告列表而不是厚信息卡流。
- 保持 `工程建设 / 政府采购 / 信息公开` 三个频道页的 UI 节奏一致。

**Non-Goals:**
- 不新增地区、金额、即将截止、更多类型等新的筛选条件。
- 不新增排序、搜索联想、热门搜索、历史记录或其它搜索功能增强。
- 不修改 API、数据模型、字段映射和页面跳转逻辑。

## Decisions

### 决策 1：搜索框成为工具区主视觉，而不是和其它控件等权
选择保留现有工具区结构，但在视觉层级上明确让搜索框成为第一关注点，tab 作为模式切换，筛选按钮作为补充操作。

原因：
- 用户对频道页的第一动作通常是先判断“搜还是筛”，搜索框必须更明显。
- 当前工具区的问题不是缺控件，而是所有控件权重太接近，导致扫读路径不明确。

备选方案：
- 继续平均缩放所有控件。
  - 否决原因：只会让整块工具区继续像一组并列组件，不能建立主次。

### 决策 2：`BidCard` 改成扫读优先的信息列表卡，而不是继续强调完整信息卡感
选择保留卡片容器，但进一步压缩卡片高度、减少行间重复，并让元信息排布更接近“标题 + 类型 + 辅助信息”三层。

原因：
- 完全去卡片化虽然能提升密度，但会破坏当前小程序整体的白卡语义。
- 保留轻卡片并强化层级，可以兼顾平台统一性和阅读效率。

备选方案：
- 改成纯列表 + 分隔线。
  - 否决原因：过于激进，会让频道页与现有详情页、收藏页风格脱节。

### 决策 3：不压缩字段来源，而是通过视觉弱化重复感
选择继续展示现有可用字段，不把“来源”强行替换成“地区”，也不伪造不存在的状态标签，而是通过更紧凑的排版和更弱的辅助信息样式来减少重复感。

原因：
- 当前数据里“来源”和“地区”并不等价，直接替换会造成语义错误。
- 这次目标是 UI 优化，不是字段模型改造。

备选方案：
- 把来源统一简化成地区。
  - 否决原因：存在信息失真风险，且依赖更稳定的地区字段支持。

## Risks / Trade-offs

- [工具区把搜索框做得过重，导致 tab 和筛选按钮被压得太弱] → 通过字号、间距和背景层级区分主次，而不是让搜索框变成另一个 CTA。
- [卡片压缩过度后，长标题可读性下降] → 保持标题最多两行和当前字重，只压缩辅助信息与卡片外壳厚度。
- [用户继续把功能建议混入 UI change] → 在 proposal/specs 中明确声明本次只做 UI 和布局，不做搜索、排序或筛选能力增强。
