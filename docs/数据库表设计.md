# 数据库表设计

本文档为系统**完整表设计**的权威定义：表清单、每张表的用途、每张表的完整字段。实现（crawler/migrations、crawler/storage、server API）须与此一致。设计依据见 openspec/changes/full-schema-spec/design.md。

说明：

- 本文档是数据库设计的**主文档**
- 旧的 notices 单表补充说明已迁移到 `docs/supplemental/存储表结构说明.md`
- 如两份文档存在重叠，以本文档为准

---

## 1. 表清单

| 序号 | 表名 | 用途简述 |
|------|------|----------|
| 1 | **notices** | 公告主表：存所有站点、所有类型的招投标/采购公告，爬虫写入、列表与详情 API 读取 |
| 2 | **dict_region** | 地区字典表：存地区编码与名称（含层级），供筛选与 notices 地区展示名称解析 |
| 3 | **dict_purchase_manner** | 采购方式字典表：存采购方式编码与名称，供筛选与 notices.purchase_manner 展示 |
| 4 | **dict_notice_category** | 公告分类字典表：存业务类型编号与展示名称，供列表 categoryName 与筛选选项 |

---

## 2. 各表用途与字段定义

### 2.1 notices（公告主表）

**用途**：存储所有站点（网站一、网站二等）、所有业务类型（招标计划、招标公告、政府采购采购公告、采购意向公开、采购公告等）的公告数据。爬虫按 (site, id) 做 upsert 写入；列表接口 /api/list 与详情接口 /api/detail/bid/:id、/api/detail/info/:id 直接或经 VIEW 读取本表，并按《接口文档-前端与小程序》做字段映射。不采用「一种类型一张表」，以便统一分页与筛选。

**唯一键**：(site, id)

**字段定义**：

| 列名 | 类型 | 可空 | 说明 |
|------|------|------|------|
| site | TEXT | 否 | 站点标识，如 site1_sc_ggzyjy、site2_ccgp_sichuan |
| id | TEXT | 否 | 业务主键，与 site 组成唯一键 |
| title | TEXT | 否 | 标题 |
| publish_time | TEXT | 可空 | 发布时间，如 2026-03-14 11:14:25 |
| info_date | TEXT | 可空 | 信息日期（网站一） |
| source_name | TEXT | 可空 | 来源/发布单位 |
| tradingsourcevalue | TEXT | 可空 | 交易来源代码（网站一，地区筛选） |
| region_name | TEXT | 可空 | 地区名称 |
| region_code | TEXT | 可空 | 地区代码，可与 dict_region 关联 |
| category_num | TEXT | 可空 | 分类/类型编号，可与 dict_notice_category 关联 |
| channel | TEXT | 可空 | 频道（网站二） |
| linkurl | TEXT | 可空 | 详情页相对路径（网站一） |
| origin_url | TEXT | 可空 | 原文绝对 URL |
| content | TEXT | 可空 | 正文或摘要（HTML/纯文本） |
| description | TEXT | 可空 | 摘要/描述（网站二） |
| open_tender_code | TEXT | 可空 | 项目编号等 |
| plan_id | TEXT | 可空 | 网站二采购公告 planId |
| budget | TEXT | 可空 | 预算金额 |
| purchase_manner | TEXT | 可空 | 采购方式，可与 dict_purchase_manner 关联 |
| open_tender_time | TEXT | 可空 | 开标时间 |
| purchaser | TEXT | 可空 | 采购人 |
| agency | TEXT | 可空 | 代理机构 |
| first_seen_at | TEXT | 可空 | 首次采集时间 |
| last_seen_at | TEXT | 可空 | 最后更新时间 |
| raw_json | TEXT | 可空 | 原始接口响应 JSON，便于排查与补采 |

**索引**：PRIMARY KEY (site, id)；idx_notices_category_publish (category_num, publish_time)；idx_notices_site (site)。

**建表脚本**：crawler/migrations/001_create_notices.sql

### 2.2 dict_region（地区字典表）

**用途**：存储地区编码与名称（含层级），数据来源为网站二 getRegionJson 或等效配置。供前端地区筛选下拉、以及将 notices.region_code / notices.tradingsourcevalue 解析为展示名称。爬虫或定时任务写入；/api/dict/regions 等接口读取。

**唯一键**：region_code

**字段定义**：

| 列名 | 类型 | 可空 | 说明 |
|------|------|------|------|
| region_code | TEXT | 否 | 地区编码，主键 |
| region_full_name | TEXT | 可空 | 地区全称，如「省本级」「成都市」 |
| parent_id | TEXT | 可空 | 父节点 id，用于树形结构 |
| is_leaf | TEXT | 可空 | 是否叶子节点，如 "1" |
| sort_order | INTEGER | 可空 | 排序序号（可选） |
| updated_at | TEXT | 可空 | 最后更新时间（可选） |

**索引**：PRIMARY KEY (region_code)；可选 INDEX(parent_id) 便于按层级查询。

**建表脚本**：crawler/migrations/002_create_dict_tables.sql

---

### 2.3 dict_purchase_manner（采购方式字典表）

**用途**：存储采购方式编码与名称，数据来源为网站二 getDictInfo（dictType 为采购方式相关）或配置。供前端采购方式筛选与 notices.purchase_manner 的展示名称。爬虫或配置写入；/api/dict/purchase-manner 等接口读取。

**唯一键**：dict_code

**字段定义**：

| 列名 | 类型 | 可空 | 说明 |
|------|------|------|------|
| dict_code | TEXT | 否 | 采购方式编码，主键 |
| dict_name | TEXT | 可空 | 采购方式名称，如「公开招标」 |
| dict_type | TEXT | 可空 | 字典类型，如 xxcg-noticeType（可选） |
| sort_order | INTEGER | 可空 | 排序（可选） |
| updated_at | TEXT | 可空 | 最后更新时间（可选） |

**索引**：PRIMARY KEY (dict_code)。

**建表脚本**：crawler/migrations/002_create_dict_tables.sql

---

### 2.4 dict_notice_category（公告分类字典表）

**用途**：存储业务类型编号（category_num / noticeType）与展示名称（如招标计划、采购意向公开、采购公告）。与 notices.category_num 对应，供列表 categoryName 与筛选选项。数据来源为静态配置或建表时初始化，与《接口文档-前端与小程序》category 取值一致。

**唯一键**：category_num

**字段定义**：

| 列名 | 类型 | 可空 | 说明 |
|------|------|------|------|
| category_num | TEXT | 否 | 分类编号，主键，如 002001009、59、00101 |
| category_name | TEXT | 可空 | 展示名称，如「招标计划」「采购意向公开」「采购公告」 |
| site_hint | TEXT | 可空 | 所属站点提示（如 site1/site2），可选 |
| sort_order | INTEGER | 可空 | 排序（可选） |
| updated_at | TEXT | 可空 | 最后更新时间（可选） |

**索引**：PRIMARY KEY (category_num)。

**初始数据建议**：002001009→招标计划；002001001→招标公告；002002001→政府采购采购公告；59→采购意向公开；00101→采购公告。

**建表脚本**：crawler/migrations/002_create_dict_tables.sql

## 3. 表间关系（逻辑）

- **notices.region_code** / **notices.tradingsourcevalue** 可关联 **dict_region.region_code**，用于解析地区展示名称。
- **notices.purchase_manner** 可关联 **dict_purchase_manner.dict_code**，用于解析采购方式名称。
- **notices.category_num** 可关联 **dict_notice_category.category_num**，用于解析 categoryName。

不强制建外键，由应用层或 VIEW 做 JOIN 或映射。
