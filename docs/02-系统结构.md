# 02-系统结构

> **作用**：说明系统如何分块、模块职责与依赖。  
> **给谁看**：开发、架构师、测试、接手人。  
> **什么时候改**：模块职责变化、服务拆分/合并、目录重构、新增关键基础设施或系统边界变化时。  
> **谁改**：本次结构变更的开发者先改；大改动由模块负责人或架构负责人补充。  
> **是否 review**：需要；架构级调整由架构负责人 review。

---

## 1. 架构图

```text
外部公告站点(site1 / site2)
          |
          v
   爬虫采集系统(crawler)
          |
          v
 SQLite 数据库(notices.db 等)
          |
          v
 统一后端服务(server / FastAPI)
      |
      +---> 微信小程序(miniapp)
      |
      +---> 运营后台前端(admin-frontend)
```

生产-like 部署（单机）：

```text
Client / Tailscale 内网
    :8000(api) / :8091(admin-frontend)
          |
          v
  api container (FastAPI + in-process dispatcher)
  admin-frontend container (static nginx)
          |
  claim queued runs --> python -m crawler.site1/site2.tasks.*
          ^
  scheduler container (supercronic + submit script)
  POST /api/admin/crawl/runs --> crawl_runs queue / history
```

## 2. 模块清单与职责

| 模块 | 职责 | 包路径/目录 |
|------|------|-------------|
| 爬虫 | 按任务抓取 site1/site2，清洗、落库、对账、增量/回填/补偿 | `crawler/`，`crawler.site1` / `crawler.site2` |
| 存储 | 公告与字典表 upsert、迁移、连接管理 | `crawler/storage.py`，`crawler/migrations/` |
| 后端 API | 列表/详情/鉴权/注册/审核/字典/管理端采集控制 | `server/main.py`，`server/` |
| 小程序 | 用户端公告展示、登录、企业认证、收藏 | `miniapp/` |
| 运营后台 | 登录、总览、企业审核、企业目录、采集控制、运行记录 | `admin-frontend/` |

---

## 3. 关键依赖关系

- **crawler** 不依赖 server；通过 `storage.get_connection` 写 SQLite，依赖 `crawler/migrations` 定义的表结构。
- **server** 依赖 crawler：认领 run 后子进程执行 `python -m crawler.site1/site2.tasks.*`，并读写同一 SQLite（`NOTICES_DB`）。
- **miniapp / admin-frontend** 仅依赖 server 的 HTTP API；不直接访问 DB 或 crawler。

---

## 4. 入口与关键基础设施

- **API 入口**：`uvicorn server.main:app`，默认 8000；健康探测用 `/openapi.json`。
- **数据库**：SQLite，路径由 `NOTICES_DB` 指定，默认 `data/notices.db`。
- **日志**：crawler 与控制面运行日志落 `logs/`、`logs/admin-crawl/` 等。
- **定时任务**：scheduler 容器内 supercronic 调用 `scheduler_submit.py`，向 API 提交 run，由 api 容器内 dispatcher 执行 crawler。

---

## 5. 约束

- API 只能单实例运行，禁止 uvicorn 多 worker。
- 必须使用整仓库镜像运行，以便 API 进程能正确调用 `crawler.*`。
- 高风险动作（如 site2 backfill --formal、cleanup）不允许由后台直接触发，需受控或拒绝。
