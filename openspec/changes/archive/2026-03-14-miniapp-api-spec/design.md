# 设计文档：miniapp-api-spec

## Context

- 仓库内《原始数据接口文档》描述的是**爬虫与数据源**（网站一、网站二）的契约，包含列表/详情的请求与响应、落库字段建议；前端与小程序不直接调用这些数据源，而是调用**自有后端**提供的 API。
- 小程序与 Web 页面（见 PAGES_OVERVIEW）需要：列表（多业务类型）、详情（招投标/信息展示）、登录、注册、审核状态、收藏等；这些能力需要对应到「后端提供给前端的接口」及其请求/响应与字段。
- 约束：接口与字段设计必须能在原始数据能力范围内实现（即爬虫可落库或可聚合的字段）；同时满足页面展示与筛选需求。

## Goals / Non-Goals

**Goals:**

- 产出一份《接口文档-前端与小程序》Markdown 文档，作为前端与小程序调用的唯一契约。
- 文档覆盖：接口清单、每接口的 Method/路径/参数/请求体/响应结构、字段定义；与首页/详情/登录/注册/审核/我的等页面的对应关系。
- 接口与字段设计有据可查：列表与详情字段可追溯到《原始数据接口文档》的落库字段；筛选参数与页面筛选区一致；登录/注册/审核/收藏等与页面表单与状态一致。

**Non-Goals:**

- 不实现后端服务或 Mock 代码；本变更只产出文档。
- 不修改《原始数据接口文档》或爬虫契约。
- 不规定后端技术栈或部署方式，仅规定对前端暴露的 API 契约。

## Decisions

### 1. 文档按「接口域」组织，并附带「页面-接口」映射

- **结构**：文档先列接口清单与适用范围，再按域分章（列表、详情、认证与用户、基础数据等），每章内每个接口包含：标识、Method、Path、请求参数/体、响应结构、字段表。文末或附录提供「页面-接口」映射表（如首页→列表接口+筛选参数；详情页→详情接口；登录页→登录/验证码接口）。
- **理由**：前端与小程序按「调哪个接口、传什么、拿什么」开发；按域组织便于后端实现与前端查找；页面映射便于验收与联调时对照。

### 2. 列表接口采用「统一列表 + 业务类型参数」或「分业务列表」二选一在文档中明确

- **建议**：统一列表 path（如 `GET /api/list` 或 `POST /api/list`），通过 query/body 参数区分业务类型（如 category / noticeType / channel），与首页一级/二级菜单对应；分页与筛选参数统一约定（page/pageSize 或 pn/rn；时间、地区、来源等筛选项）。
- **理由**：原始数据侧有网站一三类 condition、网站二两种 noticeType，后端可聚合为同一列表 API 并透传筛选，减少前端接口数量；若后端按数据源分服务，文档也可约定多 path 但响应结构统一，便于前端抽象。

### 3. 字段命名与类型与原始数据对齐，必要时做前端友好别名

- **约定**：响应字段优先使用《原始数据接口文档》中已落库字段名（如 id、title、webdate、zhuanzai、linkurl、content）；若前端需要统一命名（如 publishTime 统一 webdate/noticeTime），在文档中注明「对应原始字段 xxx」。类型（string/number/boolean）、可空、数组元素结构在文档中明确写出。
- **理由**：后端落库与原始数据一致时，可直接透传或简单映射；文档写清对应关系后，前端与后端可并行开发。

### 4. 登录/注册/审核/收藏等「自有业务接口」在文档中单独定义

- **约定**：这些接口不来自原始数据源，由自有后端提供。文档中必须列出：登录（手机号+验证码）、获取验证码、注册（企业信息+营业执照）、审核状态查询、收藏列表、收藏/取消收藏等；每项包含请求与响应契约。若当前无后端实现，可标注「Mock 约定」或「待实现」，但契约必须先定。
- **理由**：小程序 miniapp-data 规格要求按接口文档调用；先定契约可让前端 Mock 与后端实现解耦。

### 5. 文档存放位置与命名

- **建议**：仓库根或 `docs/` 下新增一份 Markdown，如《接口文档-前端与小程序.md》；或《接口文档.md》若当前不存在则可直接使用该名。本变更的 tasks 中明确产出路径，便于 miniapp-from-ui 引用。
- **理由**：与《原始数据接口文档》区分开，避免混淆；路径固定后，spec 与 README 可引用同一文件。

## Risks / Trade-offs

| 风险 | 缓解 |
|------|------|
| 原始数据缺少某页面所需字段 | 在文档中标注「依赖后端从 xxx 补充」或「仅展示已有字段」；列表/详情字段表与原始数据落库表逐项对照。 |
| 筛选参数与后端实现不一致 | 文档中列出首页各菜单组合下的全部筛选项及参数名与取值（如时间范围、地区 code、采购方式 code），与 PAGES_OVERVIEW 和原始数据接口的筛选能力一致。 |
| 登录/注册等暂无后端 | 文档仍写出完整契约；后端实现时按文档对接，前端可先 Mock。 |

## Migration Plan

- 无部署或回滚；新增文档后，miniapp-from-ui 的 miniapp-data 与前端请求层以该文档为契约实现即可。
