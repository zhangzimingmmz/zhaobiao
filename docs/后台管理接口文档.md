# 后台管理接口文档

本文档是统一运营后台的正式接口入口，覆盖企业审核、企业目录、采集控制、运行记录以及当前已识别但尚未完成的后台接口缺口。当前仓库中的极简运营后台前端工程位于 `admin-frontend/`。

## 1. 认证方式

当前后台管理接口支持以下两种管理员认证方式：

```http
Authorization: Bearer <ADMIN_TOKEN>
```

说明：

- `ADMIN_TOKEN` 由服务端环境变量配置
- 也支持通过 `POST /api/admin/login` 获取管理员 JWT
- 当前实现是最小内部鉴权方案，不是正式管理员账号体系
- 与小程序用户的 `/api/auth/login` 不是同一套认证逻辑

## 2. 接口状态说明

本文档使用以下状态标记：

- `已实现`：后端已有真实接口和代码实现
- `部分实现`：可由现有接口拼接完成，但没有独立聚合接口
- `未完成`：需求已明确，但当前后端没有独立接口实现

## 3. 后台登录相关接口

### 3.1 管理员登录

- 接口：`POST /api/admin/login`
- 状态：`已实现`

说明：

- 使用固定账号和固定密码登录
- 默认账号由 `ADMIN_USERNAME` 控制，默认值为 `admin`
- 默认密码由 `ADMIN_PASSWORD` 控制，默认值为 `admin123456`
- 登录成功后返回管理员 JWT，供前端本地长期持久化

请求体：

```json
{
  "username": "admin",
  "password": "your-password"
}
```

响应体：

```json
{
  "code": 200,
  "data": {
    "token": "admin-jwt-token",
    "tokenType": "Bearer",
    "username": "admin"
  }
}
```

### 3.2 后台总览接口

- 接口：`GET /api/admin/dashboard`
- 状态：`部分实现`

说明：

- 当前阶段不新增独立 Dashboard 聚合接口
- 当前总览页采用前端聚合策略
- 前端通过以下接口组合得到总览数据：
  - `GET /api/admin/reviews?status=pending&page=1&pageSize=1`
  - `GET /api/admin/crawl/runs?limit=1`
- 如后续需要更复杂的总览，可再补 `GET /api/admin/dashboard`

## 4. 企业审核接口

- 接口：`GET /api/admin/reviews`
- 状态：`已实现`

请求参数：

| 参数 | 类型 | 必填 | 说明 |

| page | number | 否 | 页码，从 1 开始，默认 1 |
| pageSize | number | 否 | 每页条数，默认 10，最大 100 |

        "contactName": "张三",

        "updatedAt": "2026-03-15T10:00:00Z",
        "auditAt": null,
        "auditedBy": null

## 6. 采集控制接口

### 6.1 获取可执行动作列表

- 接口：`GET /api/admin/crawl/actions`
- 状态：`已实现`

响应说明：

- 返回允许在后台触发的受控动作白名单
- 当前不允许任意脚本和任意 CLI 参数透传

支持动作：

| Action Key | Site | Kind | Notes |
|---|---|---|---|
| `site1.incremental` | site1 | routine | 无额外参数 |
| `site1.recovery` | site1 | routine | 无额外参数 |
| `site1.backfill` | site1 | maintenance | 需要 `start` / `end`，可选 `category` |
| `site2.incremental` | site2 | routine | 无额外参数 |
| `site2.recovery` | site2 | routine | 无额外参数 |
| `site2.backfill` | site2 | maintenance | 需要 `start` / `end`，可选 `noticeType` |
| `site2.precheck` | site2 | diagnostic | 无额外参数 |
| `site2.reconcile` | site2 | diagnostic | 需要 `start` / `end` |

明确拒绝的动作：

- `site2.cleanup`
- `site2.backfill-formal`
- `site2.reconcile.verify-recovery`
- `site2.reconcile.verify-boundary`

### 6.2 提交采集运行请求

- 接口：`POST /api/admin/crawl/runs`
- 状态：`已实现`

请求体：

```json
{
  "actionKey": "site2.backfill",
  "params": {
    "start": "2026-03-01",
    "end": "2026-03-07"
  }
}
```

成功响应示例：

```json
{
  "code": 200,
  "message": "请求已受理",
  "data": {
    "id": "run-id",
    "site": "site2",
    "actionKey": "site2.backfill",
    "status": "queued"
  }
}
```

说明：

- 请求是异步的
- 若站点已有 `queued` 或 `running` 的后台任务，请求会被拒绝并记录为 `rejected`

## 7. 运行记录接口

### 7.1 获取运行记录列表

- 接口：`GET /api/admin/crawl/runs`
- 状态：`已实现`

请求参数：

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| status | string | 否 | 运行状态 |
| site | string | 否 | 站点标识 |
| actionKey | string | 否 | 动作键 |
| limit | number | 否 | 默认 50，最大 200 |

说明：

- 当前列表接口返回 `list`
- 主要字段包括 `id`、`site`、`actionKey`、`status`、`statusReason`、`summary`、`requestedAt`

### 7.2 获取运行详情

- 接口：`GET /api/admin/crawl/runs/{run_id}`
- 状态：`已实现`

当前详情可返回的主要字段：

- `id`
- `site`
- `taskName`
- `actionKey`
- `runKind`
- `requestedBy`
- `status`
- `statusReason`
- `requestPayload`
- `resultPayload`
- `summary`
- `requestedAt`
- `startedAt`
- `finishedAt`
- `logPath`
- `command`
- `exitCode`
- `fetchedCount`
- `upsertedCount`
- `errorCount`

说明：

- 详情页已经能支撑当前极简后台查看动作、参数、执行摘要和失败原因
- 更完整的日志展示能力仍可后续增强

## 8. 权限与约束说明

- 所有 `/api/admin/*` 接口都要求有效管理员 token
- 普通用户 token 不能访问管理员接口
- 当前后台仍然是最小内部使用模式，不是正式多角色系统
- 采集控制只允许受控白名单动作
- 后台请求不会直接执行任意 Python 模块或任意 CLI 参数
