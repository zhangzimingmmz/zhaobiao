## Context

当前工程建设频道已经在小程序中暴露“招标文件预公示”页签，并以 `002001010` 作为分类参数请求 `/api/list`，但系统内部仍停留在旧的三类 site1 采集模型。现状表现为：

- `crawler/site1/config.py` 仅配置 `002001009`、`002001001`、`002002001`
- site1 的 backfill、incremental、recovery 都遍历 `config.ALL_CATEGORY_IDS`，因此不会触达 `002001010`
- 后端 `CATEGORY_NAMES` 和 `dict_notice_category` 初始化数据也没有 `002001010`
- `/api/list` 对 `category` 做精确匹配，前端传入新分类时只能得到空结果

这次变更横跨 miniapp、crawler、SQLite 字典初始化和 notices API，属于已有能力的端到端补齐，而不是单点修复。

## Goals / Non-Goals

**Goals:**

- 让 site1 例行采集任务将“招标文件预公示”作为正式分类纳入增量、补偿和回填流程
- 让 notices API、分类字典和前端展示对 `002001010` 保持一致命名和查询行为
- 提供可验证的上线与回填路径，避免再次出现“前端有入口、后端无数据”的状态

**Non-Goals:**

- 不在本变更中新增新的站点、调度器能力或后台手工控制入口
- 不重构 site1 的整体抓取架构，只扩充现有分类配置与映射
- 不承诺源站一定已有稳定数据；若源站分类码与预期不符，本变更只要求通过验证流程及时发现并修正

## Decisions

### 1. 将 `002001010` 视为 site1 的正式第四类公告

直接在 `crawler/site1/config.py` 的 `CATEGORIES` 中增加 `002001010`，并让 `ALL_CATEGORY_IDS` 自动带入 backfill、incremental、recovery。这样能够复用现有时间窗口、分页和详情补齐逻辑，改动面最小，也不会引入新的任务入口。

备选方案：

- 单独为“招标文件预公示”写一条专用任务链路。未采用，因为它与现有 site1 分类处理模式重复，后续维护成本更高。
- 继续由前端复用 `002001009` 或 `002001001`。未采用，因为这会把不同业务类型混在一起，无法保证筛选和命名准确。

### 2. 统一把分类契约收敛到同一组常量与字典数据

实现层需要同时更新 crawler 配置、后端 `CATEGORY_NAMES` 和 `dict_notice_category` 初始数据，确保采集落库、API 查询和前端展示使用同一个分类编号与名称。这样可以避免“能采到但接口不认”“接口认但字典不返回”这类半接通状态。

备选方案：

- 只更新 crawler 和 API，不改字典表。未采用，因为字典接口会继续缺少该分类，前端和后台的筛选源会不一致。

### 3. 用回填验证真实分类有效性，而不是只依赖代码静态接线

新增分类后，需要通过一次受控 backfill 或限定窗口的增量验证，检查数据库是否真的出现 `002001010` 记录。这样可以尽早发现源站真实 `categorynum` 与前端约定不一致的情况。

备选方案：

- 只做代码改动，不做数据分布验证。未采用，因为这次问题的本质就是“代码有入口但链路没打通”，没有实际落库验证就无法关闭风险。

## Risks / Trade-offs

- [源站分类码与 `002001010` 不一致] → 先按当前前端约定接线，但在验收中必须检查实际落库分布；若探测结果为 0，需要及时修正 source category 配置
- [旧环境已执行过字典迁移，新增初始数据不会自动补齐] → 通过新增 migration 或幂等补数逻辑确保已有库能写入 `dict_notice_category`
- [新增分类后回填窗口过大] → 先用单日或小窗口验证，再决定是否对历史全量回填
- [前端收藏、筛选或空态逻辑仍遗漏新分类] → 实现时同步检查 `miniapp` 中分类归属和展示兜底逻辑

## Migration Plan

1. 代码层补充 `002001010` 的 crawler 配置、API 映射和分类字典
2. 在本地或测试库执行 schema 初始化，确认 `dict_notice_category` 含 `002001010`
3. 先运行受控窗口的 site1 backfill 或 incremental，确认 notices 中出现 `category_num='002001010'`
4. 部署后观察例行任务结果与工程建设频道列表是否返回数据
5. 若出现异常，可回滚代码；数据库中新增的字典项为向后兼容数据，不影响旧分类查询

## Open Questions

- 源站 site1 上“招标文件预公示”的真实 `categorynum` 是否稳定为 `002001010`
- 历史数据是否需要一次性补齐，还是仅依赖增量和最近 48 小时补偿即可满足业务上线
