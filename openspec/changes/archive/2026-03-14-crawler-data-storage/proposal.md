# Proposal: 爬虫数据存储与 API 查询表设计

## Why

项目已有《原始数据接口文档》（网站一/网站二各接口的请求与响应、落库字段建议）和《接口文档-前端与小程序》（列表/详情 API 的请求参数与响应字段）。爬虫落库与前端查询之间缺少明确的数据存储设计：原始数据应如何存、是否单独建「供 API 查询」的表、按业务类型分多张表还是统一表，尚未定稿。若不先定表结构与读写边界，后续爬虫落库与 API 实现容易反复调整。本变更为爬虫数据定义一套可落地的存储与查询表方案，并在文档中固定下来，供爬虫与 API 实现共同遵守。

## What Changes

- **明确原始数据存储策略**：约定爬虫写入的「原始表」结构（字段来源《原始数据接口文档》落库建议）、唯一键（如 site + id）、是否按站点或按业务类型分表、是否保留 raw_json 等。
- **明确 API 查询数据来源**：在「仅用原始表供 API 查询」与「原始表 + 独立 API 查询表」两种方案中做选择并给出理由；若采用独立查询表，约定其字段与《接口文档-前端与小程序》列表/详情字段的映射关系及更新方式（同步/视图/ETL）。
- **统一多数据源下的表形态**：网站一（三类公告）、网站二（采购意向公开、采购公告）及后续可能新增站点，在「一种类型一张表」与「统一公告表 + 类型/站点字段」之间做选择，并说明取舍与扩展方式。
- **产出可执行的表结构说明**：供实现爬虫 storage 与公告 API 时直接参照，避免各做各的 schema。

无破坏性变更：本次仅产出设计与规范文档，不强制修改已有代码；已有 site1-crawler-and-ui 的 storage/API 实现可在后续按本方案迁移或对齐。

## Capabilities

### New Capabilities

- **crawler-storage-schema**：爬虫原始数据与 API 查询层的数据表设计规范。包含：原始数据表结构（字段、唯一键、分表策略）、是否单独建 API 查询表及其与前端接口字段的对应关系、多站点/多类型下的表数量与命名约定（一类型一表 vs 统一表），以及和《原始数据接口文档》《接口文档-前端与小程序》的对照说明。

### Modified Capabilities

- （无。notices-data-api 的需求「数据来自爬虫落库、字段与 UI 对齐」不变，本变更仅明确落库与查询所依循的表结构，不改变接口契约本身。）

## Impact

- **文档与规范**：新增或更新 crawler 与 API 相关的数据表设计文档（位于 openspec 本变更的 design / specs 中），可同步到项目 wiki 或 crawler/ 下 README。
- **实现**：后续 crawler/site1、crawler/site2 的 storage 模块及公告列表/详情 API 的实现将按本方案建表与读写；若当前已有原型表，需在实现阶段按本方案做一次对齐或迁移。
- **依赖**：无新增运行时依赖；仅约定存储层形态（如 SQLite/PostgreSQL 表结构），与现有技术选型兼容。
