# 统一运营后台需求清单

## 1. 文档目的

本文档用于明确当前项目“统一运营后台”的第一阶段需求边界，整理后台必须具备的核心模块、页面、关键功能与优先级，作为后续新建 `admin-frontend/` 工程的产品蓝图。

这份文档关注的是“后台应该有什么”，不要求所有接口现在都已经存在。对于缺失 API 的部分，可以先按占位态、只读态或待补接口状态规划页面。

## 2. 项目背景

当前仓库已经形成以下正式系统边界：

- `crawler`：负责招投标与采购信息采集
- `server`：负责业务接口、管理接口、认证与采集控制
- `miniapp`：负责用户侧展示、登录、认证和收藏

目前的问题不是“后台要不要”，而是“后台前端没有一个正式、清晰、可持续扩展的工程边界”。因此需要新增一个正式的统一运营后台，用于承接企业审核、企业目录、采集控制、运行记录和后台总览等能力。

## 3. 建设目标

统一运营后台第一阶段目标：

- 提供统一入口和统一导航
- 支撑运营审核人员处理企业认证申请
- 支撑运营人员查看企业目录
- 支撑内部人员发起采集任务和查看运行记录
- 提供最小可用的总览页

当前不纳入第一阶段范围的内容：

- 复杂 RBAC 权限体系
- 完整用户管理中心
- BI 报表中心
- 消息中心
- CRM 式客户运营台账
- 客服系统

## 4. 后台信息架构

建议统一运营后台第一阶段采用如下一级导航：

```text
统一运营后台
├─ 总览
├─ 企业审核
├─ 企业目录
├─ 采集控制
└─ 运行记录
```

说明：

- “总览”用于看最基本的待处理事项和采集状态
- “企业审核”用于处理认证申请任务流
- “企业目录”用于查看当前企业存量状态
- “采集控制”用于发起允许的采集动作
- “运行记录”用于查看历史运行和执行详情

## 5. 核心模块需求清单

### 5.1 后台登录与访问控制

| 系统 | 页面/模块 | 必须功能 | 可后补功能 | 依赖接口 | 优先级 |
|---|---|---|---|---|---|
| 统一运营后台 | 登录页 | 提供一个正式后台登录页，支持输入固定账号和固定密码 | 登录页品牌化、登录公告 | 需要最小登录接口或固定凭证校验机制 | P0 |
| 统一运营后台 | 登录校验 | 校验账号密码，成功后进入后台首页，失败后给出明确提示 | 登录错误次数限制 | 需要后台登录接口 | P0 |
| 统一运营后台 | 路由访问控制 | 未登录时禁止访问后台页面，访问后台路由时跳转登录页 | 登录后回跳原目标页 | 依赖登录态校验 | P0 |
| 统一运营后台 | 本地登录态持久化 | 登录成功后将登录态保存在本地浏览器中，默认长期有效 | 更换更安全的会话策略 | 可先采用本地存储机制 | P0 |

说明：

- 第一阶段按最小方案处理，只支持一个固定账号、一个固定密码
- 当前默认只有一个运营人员使用，不做多账号和角色体系
- 登录成功后默认长期有效，除非主动清理浏览器缓存
- 第一阶段不做退出登录、会话过期、找回密码、短信登录、邮箱登录、验证码和复杂权限管理

### 5.2 后台基础壳层

| 系统 | 页面/模块 | 必须功能 | 可后补功能 | 依赖接口 | 优先级 |
|---|---|---|---|---|---|
| 统一运营后台 | 基础布局 | 提供一个稳定的后台页面容器，包含菜单区和内容区 | 视觉优化 | 无 | P0 |
| 统一运营后台 | 简单导航 | 支持跳转到总览、企业审核、企业目录、采集控制、运行记录 | 导航高亮优化 | 无 | P0 |
| 统一运营后台 | 页面标题区 | 每个页面显示当前页面名称和基本说明 | 面包屑 | 无 | P0 |
| 统一运营后台 | 基础状态反馈 | 提供加载中、空数据、接口异常等基础页面提示 | 更细粒度状态组件 | 无 | P0 |

### 5.3 总览 Dashboard

| 系统 | 页面/模块 | 必须功能 | 可后补功能 | 依赖接口 | 优先级 |
|---|---|---|---|---|---|
| 统一运营后台 | 总览首页 | 展示后台最基本的汇总信息，作为默认进入页 | 后续补更多卡片 | 需聚合多个后台接口，部分可先占位 | P0 |
| 统一运营后台 | 待审核数量 | 显示待审核企业数量，并支持进入审核列表 | 更复杂提醒 | `/api/admin/reviews` 或等价接口 | P0 |
| 统一运营后台 | 采集运行概况 | 显示最近一次采集状态或最近运行情况 | 趋势图和异常中心 | 现有采集运行接口，部分字段可能缺失 | P0 |

### 5.4 企业审核

| 统一运营后台 | 运行详情页 | 查看动作、参数、执行摘要和失败原因 | 完整日志能力 | `/api/admin/crawl/runs/{id}` 或等价接口 | P0 |

## 6. 第一阶段必须落地的页面

第一阶段最少应包含以下页面：

1. 后台入口页或登录页
2. 总览页
3. 企业审核列表页
4. 企业审核详情页
5. 企业目录页
6. 采集控制台页
7. 运行记录列表页
8. 运行详情页

说明：

- 如果某些详情数据暂时没有 API，可以先以结构化占位态落地
- 只要页面是“后台工作流中必须存在”的，就应该先定义出来

## 7. 第一阶段必须具备的运营闭环

后台第一阶段至少要支持两个闭环：

### 7.1 企业审核闭环

```text
查看待审核列表
   ↓
进入审核详情
   ↓
查看企业资料和营业执照
   ↓
通过 / 驳回
   ↓
企业目录可见最新状态
```

### 7.2 采集运营闭环

```text
查看总览或采集控制台
   ↓
选择采集动作并填写参数
   ↓
提交运行请求
   ↓
查看运行记录
   ↓
进入运行详情
   ↓
查看摘要 / 失败原因
```

## 8. API 缺失时的处理原则

为了避免“等接口齐了再做页面”，第一阶段明确采用以下策略：

- 页面可以先存在
- 缺失接口的模块可以先显示占位态
- 详情字段不足时可以先做只读结构页
- 缺失接口要被记录下来，后续作为服务端补齐清单

也就是说，后台规划优先级高于接口完备度。

## 9. 优先级总结

### P0：必须有

- 后台登录页
- 固定账号密码登录
- 路由访问控制
- 本地长期登录态
- 全局布局和导航
- 总览页基础结构
- 企业审核列表
- 企业审核详情
- 审核通过/驳回
- 企业目录
- 采集控制台
- 运行记录列表
- 运行详情页
- 全局加载/空状态/错误态

### P1：建议第一阶段尽量补齐

- 企业目录简单搜索
- 更完整的采集状态信息
- 更完整的运行详情信息

### P2：后续扩展

- 复杂角色权限体系
- 趋势图与报表
- 告警中心
- 企业详情页
- 审核操作审计

## 10. 建议的下一步

建议按以下顺序推进：

1. 在仓库中新增正式后台工程目录，例如 `admin-frontend/`
2. 先完成后台基础壳层和一级导航
3. 先接入已有的企业审核、企业目录、采集控制、运行记录接口
4. 页面先跑通主工作流，再补齐缺失字段和缺失接口
5. 将接口缺口整理成下一轮 change 或开发任务
