## Context

当前小程序收藏完全依赖本地存储: 页面通过 `miniapp/src/utils/favorites.js` 写入 `favorites_records` 和 `favorites_selected_type`, 后端没有正式收藏表和收藏接口。与此同时, 用户体系、JWT 鉴权、公告与文章详情接口都已经存在, 因此收藏具备切换到账号级服务端持久化的基础。

本次设计的边界已经明确:

- 项目尚未正式上线, 不需要兼顾历史本地收藏迁移。
- 未登录用户不允许收藏。
- 收藏页不保留失效快照; 当源公告或文章消失时, 收藏页对应项直接消失。

这意味着收藏应建模为“用户与目标内容的关系”,而不是“本地展示副本”。

## Goals / Non-Goals

**Goals:**

- 新增登录用户的服务端收藏关系存储与查询能力。
- 让小程序首页、详情页、收藏页以服务端收藏状态为唯一事实来源。
- 对公告(`bid`)与文章(`info`)统一提供收藏切换与列表查询。
- 保证失效源数据不会继续出现在收藏页中。

**Non-Goals:**

- 不保留游客本地收藏。
- 不为收藏记录保留展示快照或软删除历史。
- 不处理历史本地收藏向服务端批量迁移。
- 不在本次设计中新增推荐、最近收藏、收藏搜索等增强功能。

## Decisions

### 1. 使用关系表而非快照表

服务端新增收藏关系表,仅保存当前用户、收藏对象标识和创建时间。收藏列表查询时再实时 join `notices` 或 `articles` 返回展示字段。

建议字段:

- `id`
- `user_id`
- `target_type` (`bid` | `info`)
- `target_id`
- `target_site` (`bid` 推荐必填, `info` 可空)
- `created_at`

选择理由:

- 用户尚未上线, 无需沿用本地快照结构。
- 源数据消失时, 收藏项也应消失, 因此没有必要保存长期展示副本。
- 实时 join 可确保标题、时间、来源等字段与主数据一致。

备选方案:

- 保存收藏快照: 否决,因为与“源数据消失则收藏消失”的要求冲突,且会增加同步复杂度。

### 2. `bid` 收藏关系保留 `target_site`

尽管当前详情接口主要按 `id` 查询, 但 `notices` 的数据库主键模型本质上是 `(site, id)`。因此服务端收藏表对 `bid` 类型保留 `target_site`, 防止未来站点间 `id` 冲突。

备选方案:

- `bid` 仅存 `target_id`: 否决,因为会把当前实现上的简化查询缺陷固化到数据模型里。

### 3. 收藏 API 以“切换 + 列表 + 状态”三类能力为中心

最小后端接口集合建议为:

- `GET /api/favorites`
- `POST /api/favorites/toggle`
- `GET /api/favorites/status` 或在详情接口中内联 `favorited`

其中 `toggle` 负责心形点击, `list` 负责收藏页, `status` 负责详情/列表初始状态。是否把 `favorited` 直接并入详情响应可在实现阶段决定, 但至少要有稳定的状态来源。

备选方案:

- 仅实现 `toggle` 和 `list`: 可行,但前端会更难初始化详情页心形状态。

### 4. 未登录用户直接被拦在收藏动作之前

未登录状态下仍可看见收藏入口, 但触发收藏动作时必须得到明确的登录引导, 而不是落回本地存储。

备选方案:

- 未登录隐藏心形: 可行,但会让列表与详情布局因登录态变化而分叉; 本设计优先选择统一视觉结构。

### 5. 失效收藏在列表阶段直接过滤

`GET /api/favorites` 查询用户收藏关系后, 只返回仍能 join 到有效 `notices` / `articles` 的记录。这样前端不需要额外处理“已失效”卡片。

备选方案:

- 返回失效壳子并提示“内容已删除”: 否决,因为当前产品已明确要求消失即可。

## Risks / Trade-offs

- [Risk] 详情页与列表页如果分别请求状态, 会增加接口次数 → Mitigation: 实现阶段优先评估在详情接口内联 `favorited` 的方案。
- [Risk] `bid` 收藏若前端没有稳定传递 `site`, 服务端无法唯一定位 → Mitigation: 在前端收藏动作中显式补齐 `site` 字段, 并为缺失情况返回参数错误。
- [Risk] 收藏列表实时 join 可能在将来数据量增大后带来性能压力 → Mitigation: 先通过必要索引满足当前规模, 将快照优化留到后续。
- [Risk] 将游客收藏彻底移除后, 用户第一次使用时会感知门槛提升 → Mitigation: 在未登录收藏动作处提供直接登录引导, 而不是静默失败。

## Migration Plan

1. 新增收藏关系表与必要唯一索引。
2. 实现用户端收藏接口, 并补充鉴权校验。
3. 小程序详情页、首页卡片、收藏页改为使用服务端收藏数据。
4. 下线本地收藏数据作为主数据源的逻辑, 仅保留必要的 UI 选择状态。
5. 通过真机验证未登录收藏拦截、已登录收藏切换、源数据删除后的收藏过滤行为。

## Open Questions

- 详情页初始 `favorited` 状态最终由独立接口返回, 还是并入详情接口响应更合适。
- `GET /api/favorites` 是否需要分页, 还是当前阶段直接返回全部列表即可。
