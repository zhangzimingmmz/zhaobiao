## Context

当前 `工程建设 / 政府采购 / 信息公开` 三个频道页已经完成了功能拆分，但视觉结构仍然保留了较厚的旧列表页习惯：二级 header 与控制区之间的过渡感不够轻，控制区仍像一整块白色面板，`SecondaryTabs`、搜索框、筛选按钮和 `BidCard` 的高度都偏大，导致页面呈现出“层多、块厚、节奏松”的观感。与此同时，用户实际看到的部分截图还混入了旧缓存版本里的双标题 / 双返回形态，因此新的对稿标准必须明确要求先以最新构建为准。

这次变更只聚焦频道页密度与头部链路，不调整数据逻辑、筛选参数、列表接口或路由关系。相关实现主要集中在 `miniapp/src/styles/channel-page.scss`、三个频道页的 `index.tsx`、以及复用组件 `SecondaryTabs`、`FilterBar`、`BidCard` 的样式层。

## Goals / Non-Goals

**Goals:**
- 将频道页统一收敛为“单一二级头部 + 轻工具带 + 更密的信息列表”。
- 明确禁止频道页中出现重复标题、重复返回语义或额外的大标题层。
- 将控制区从厚重面板改成轻工具带，压缩 tab、搜索框、筛选按钮和列表卡片的垂直密度。
- 保持现有搜索、筛选、收藏、分页和列表数据行为不变。

**Non-Goals:**
- 不重构首页、登录页、注册页或审核状态页。
- 不新增搜索功能、筛选维度或新的频道目的地。
- 不修改 API、数据模型或频道页内部业务逻辑。

## Decisions

### 决策 1：频道页只保留单一 secondary header，不再叠加页面内大标题
选择以 `TopBar variant="secondary"` 作为频道页唯一头部链路，正文区不再允许渲染第二套频道标题、第二个返回箭头或额外的标题栏。

原因：
- 用户反馈中的“重复感”核心来源就是头部链路像堆了两层，哪怕部分截图来自旧缓存，也必须在规范上明确禁止这种形态。
- 二级信息列表页最应该突出的是工具区和列表本身，而不是反复重复页面标题。

备选方案：
- 保留正文区大标题，只弱化顶栏标题。
  - 否决原因：这会继续制造“头部结束后又来一套标题”的观感，不利于密度收敛。

### 决策 2：控制区改为轻工具带，而不是继续保留完整白色面板卡
选择保留 tab、搜索和筛选的组合关系，但通过削弱外层容器的卡片感、减少 padding 和圆角、压缩控件高度，把控制区表达为工具带而不是一张完整的面板卡。

原因：
- 当前最大空白感和厚重感都来自控制区外壳过重。
- 用户真正需要的是“筛选工具可见”，而不是一块和列表卡片同权重的大白卡。

备选方案：
- 完全去掉外层容器，让 tab、搜索和筛选裸排。
  - 否决原因：会让页面结构过散，工具区与列表之间缺少聚合关系。

### 决策 3：`BidCard` 在频道/收藏列表中采用更紧的列表密度
选择继续复用 `BidCard` 组件，但将频道/收藏这类列表上下文下的卡片表现压缩为更接近“列表卡”的密度：标题、标签、来源和发布时间的间距收紧，收藏心形缩小，减少卡片上下 padding。

原因：
- 这些页面展示的信息字段很少，如果继续沿用偏厚的信息卡节奏，一页内容量会过低。
- 复用同一组件可以保持卡片家族统一，不必再拆出第二套频道卡组件。

备选方案：
- 单独为频道页新建一个更薄的卡片组件。
  - 否决原因：会引入额外组件分叉，而问题本质只是密度和层级，不是字段模型差异。

### 决策 4：对稿验收以最新构建为准，并明确缓存不是设计基线
选择在变更中显式要求先使用最新构建截图验收，再讨论头部重复问题。

原因：
- 当前用户截图与现有源码已不一致，说明微信开发者工具缓存会直接干扰样式验收。
- 如果不把这一点写入设计和任务，后续会反复对着旧截图修不存在的结构问题。

## Risks / Trade-offs

- [控制区弱化过度导致筛选区不够显眼] → 保留轻边框和组块关系，只削弱“卡片感”而不是让结构完全消失。
- [`BidCard` 全局压缩后影响其他页面阅读性] → 仅按频道/收藏等列表上下文校正卡片密度，不改变详情页或特殊状态卡。
- [用户继续拿旧缓存截图做对稿] → 在任务中明确加入“最新构建截图核对”作为最后一道验收步骤。
