# 05-数据库设计

> **作用**：给人提供「数据库业务视角」的入口，不替代 migration/schema。  
> **给谁看**：后端、测试、DBA、数据分析、新人。  
> **什么时候改**：新增/删除核心表、表关系、状态字段语义、关键约束或存储策略变化时。  
> **谁改**：改表结构的人先改。  
> **是否 review**：需要；与数据库变更代码同 PR review。

---

## 1. 核心表清单

| 表名 | 职责 |
|------|------|
| **notices** | 公告主表：所有站点、类型的招投标/采购公告；爬虫 upsert，列表与详情 API 读取；(site, id) 唯一。 |
| **dict_region** | 地区字典：地区编码与名称（含层级），供筛选与 notices 地区展示。 |
| **dict_purchase_manner** | 采购方式字典：编码与名称，供筛选与 notices.purchase_manner 展示。 |
| **dict_notice_category** | 公告分类字典：业务类型编号与展示名称，供列表 categoryName 与筛选。 |
| **users** | 用户表：手机号等，登录与关联企业认证。 |
| **enterprise_applications** | 企业认证申请：企业信息、执照、状态、审核人与时间。 |
| **crawl_runs** | 采集运行记录：任务类型、状态、时间、日志路径等（具体以 migration 为准）。 |
| **articles** | 文章表：运营人工编辑的图文信息元数据（标题、摘要、封面、公众号链接、分类、状态等）。 |
| **article_operation_logs** | 文章操作日志：记录创建、更新、发布、下线、删除等操作。 |

---

## 2. 表关系简述

- **notices**：独立主表；region_code / tradingsourcevalue 可关联 dict_region；category_num 可关联 dict_notice_category；purchase_manner 可关联 dict_purchase_manner。
- **enterprise_applications**：通过 user_id 关联 users。
- **crawl_runs**：与 notices 无直接外键；逻辑上记录每次采集任务的执行结果。
- **articles**：独立主表；存储公众号文章链接与元数据；wechat_article_url 唯一。
- **article_operation_logs**：通过 article_id 关联 articles。

---

## 3. 核心状态字段

- **enterprise_applications.status**：如 pending / approved / rejected；审核后 audited_by、audit_at、reject_reason 等留痕。
- **crawl_runs**：run 状态（如 queued、running、success、failure）以实际 migration 和代码为准。
- **articles.status**：draft / published；publish_time 为发布时间。

---

## 4. 主键、唯一约束与关键索引

- **notices**：PRIMARY KEY (site, id)；常用索引如 (category_num, publish_time)、(site)。
- **dict_***：各字典表主键见 crawler/migrations。
- **users**：主键 id，mobile 唯一。
- **enterprise_applications**：主键 id，user_id 等见 migration。
- **articles**：主键 id；wechat_article_url 唯一；索引 (status, publish_time)、(category)。
