# Proposal: 完整表设计规范（表清单、用途、字段）

## Why

当前表设计分散在《存储表结构说明》与多个变更（crawler-data-storage、storage-dict-tables）中，缺少一份**单一权威文档**，明确回答：系统**到底有哪些表**、**每张表的作用是什么**、**每张表对应的字段是什么**。需求方希望得到「所有表都设计好」的可验收结果，且这些内容应在**需求与规范层面**写清楚，而非仅存在于实现或零散说明中。本变更产出并固化**完整表设计**：表清单、每表用途、每表完整字段定义，作为爬虫、API 与前端对接的单一依据。

## What Changes

- **新增《数据库表设计》文档（或等价规范）**：包含（1）**表清单**：系统内所有业务表名称；（2）**每张表的用途**：该表存什么、谁写入、谁读取、与哪些 API/业务对应；（3）**每张表的完整字段定义**：列名、类型、可空、说明、来源/约束。覆盖公告表 `notices` 与所有字典表（地区、采购方式、公告分类）。
- **在需求中明确约定**：通过 spec 要求「系统 SHALL 提供完整表设计，包含表清单、每表用途、每表字段」；实现可通过一份文档或 schema 文件满足，便于验收与后续维护。
- **与现有 migrations 与 API 对齐**：表设计文档与 crawler/migrations 下建表 SQL、server 与 crawler/storage 的读写保持一致；若有差异以表设计文档为权威定义并同步迁移脚本。

无破坏性变更：不改变现有 notices 或字典表的业务语义，仅将「有哪些表、用途、字段」集中定义并纳入需求。

## Capabilities

### New Capabilities

- **full-schema-doc**：完整表设计规范。系统 SHALL 提供并维护一份表设计文档（或纳入 docs/存储表结构说明 的扩展），其中 SHALL 包含：**表清单**（所有表名）；**每张表的用途说明**（存储内容、读写方、关联业务）；**每张表的完整字段定义**（列名、类型、可空、说明）。每张表 MUST 在该文档中具备明确的用途描述与完整列定义，以便实现与验收有据可查。

### Modified Capabilities

- （无。本变更为新增规范与文档，不修改既有 spec 的接口或存储行为要求。）

## Impact

- **文档**：新增或大幅扩展《数据库表设计》或《存储表结构说明》，形成「表清单 + 每表用途 + 每表字段」的单一来源；design 与 spec 中可直接引用或内联该结构。
- **实现**：建表 SQL（migrations）与 crawler/storage、server 的列名/类型与表设计文档一致；新表（如字典表）按文档补充迁移与代码。
- **验收**：可通过「是否存在表设计文档」「文档是否包含全部表、每表是否有用途与字段表」进行验收。
