## Context

当前小程序经过数轮调整后，功能链路已经比较完整：

```text
launch → login → home entry → channel page → detail
```

问题不再是功能缺失，而是视觉表达没有对齐参考 PDF。当前实现存在几个稳定问题：

- 首页入口页 hero 区过重，像宣传页而不是业务入口页。
- 频道页顶部、筛选面板和列表卡片都偏“卡片套卡片”，层级太多，组件感太重。
- 登录页已经具备 form + preview 的正确结构，但 still 偏标准表单页，不够收敛。
- 详情页仍然是“通用大白卡详情页”表达，没有形成更接近 PDF 的业务详情层级。

同时，OpenAI 官方关于“designing delightful frontends with GPT-5.4”的指导也指出：如果视觉护栏不明确，模型会回落到高频默认 UI 模式，例如大渐变 hero、厚卡片、多个竞争色块和 dashboard 式拼装。当前项目的偏差与这种默认回落高度一致。

因此，这次 change 的核心不是重新发明视觉方向，而是先把“什么不能再出现、什么必须收敛”说清楚，再围绕现有页面家族逐步收口。

## Goals / Non-Goals

**Goals:**

- 为首页入口页、登录页、频道页、详情页建立一套统一的视觉收敛规则。
- 让页面整体回到更像原生小程序业务页的表达：白底、轻蓝强调、清晰标题层级、较少装饰性视觉竞争。
- 明确页面家族差异：
  - 一级页：简洁品牌壳与业务入口
  - 频道页：工具条 + 列表页
  - 次级页：返回 + 内容上下文
- 把 PDF 对稿要求转化为可验证的样式和结构约束，减少后续改动再次跑偏。

**Non-Goals:**

- 不改现有登录门禁、筛选逻辑、收藏逻辑或详情数据契约。
- 不通过 redesign 引入新的业务模块、运营横幅、动效框架或可视化配置平台。
- 不要求与 PDF 做像素级 1:1 复刻；重点是页面气质、层级、密度和结构对齐。
- 不把整套小程序扩展成“重品牌营销式设计”，避免走向过度表现化。

## Decisions

### 1. 先定义视觉护栏，再改页面

**选择**：本次变更首先定义一套视觉负面清单和正向约束，再据此调整页面。

**核心护栏：**

- 避免大面积装饰性渐变作为常规业务页背景。
- 避免“外层卡片 + 内层卡片 + 行内标签卡片”层层嵌套。
- 避免多个高饱和色同时竞争。
- 避免把 routine product UI 做成 landing page hero。
- 用 typography、留白、边框轻重来建立层级，而不是靠堆更多容器。

**理由**：

- 当前问题不是单个组件失真，而是整套页面共同掉进了泛化 UI 模式。
- 只有先定义护栏，后续 AI 或开发才能稳定产出与 PDF 更接近的页面。

**替代方案**：逐页微调，不总结视觉规则。

- 被否决原因：容易导致不同页面各自“变好一点”，但整体语言仍然不统一。

### 2. 首页入口页采用“简洁品牌头部 + 频道入口卡”模型

**选择**：首页入口页保留品牌名称，但显著降低 hero 区高度和宣传感，把首屏重心让给三大频道入口。

**具体方向：**

- 头部保持白底或极浅蓝底，不再使用高存在感蓝色 hero。
- 品牌标题保留，但副标题与装饰性文案弱化。
- 三张频道入口卡使用统一家族，减少“左色条 + 大字牌 + 箭头”竞争。
- 页面留白向上收拢，避免“上重下空”。

**理由**：

- PDF 的首页入口页本质是业务入口，而不是营销首屏。
- 当前实现中，hero 与入口卡同时争主角，导致首屏注意力分散。

**替代方案**：继续保留高饱和 hero，仅微调卡片。

- 被否决原因：无法根治首页“宣传页感”。

### 3. 频道页家族统一采用“白底头部 + 薄控制区 + 紧凑列表”模型

**选择**：把工程建设、政府采购、信息公开视为同一频道页家族，统一视觉结构。

**统一结构：**

```text
白底返回头部
↓
页面标题 / 局部上下文
↓
轻量控制区（tab + 搜索 + 两个筛选）
↓
紧凑列表卡片
```

**要求：**

- 头部不再使用整条蓝底 tab 页样式。
- 控制区变薄、变扁，减少单独容器感。
- 搜索框和筛选按钮高度降低。
- 列表卡片内容节奏压缩，弱化阴影，减少浮层感。

**理由**：

- 频道页是 PDF 对稿中的核心业务页，当前最大偏差恰好出现在顶部和控制区。
- 统一家族后，三类频道页可以共享样式和验收规则。

**替代方案**：分别调整三个频道页。

- 被否决原因：会增加维护成本，并放大风格漂移风险。

### 4. 登录页继续 form-first，但预览区必须次级化

**选择**：登录页保留当前 form + preview 的结构，但进一步收敛成“表单主导、预览辅助”的次级页。

**具体方向：**

- 标题区和说明区更紧凑，强调登录主语义。
- 登录表单卡减少不必要空隙。
- 预览区上边距收紧、边框更轻、内部轮播卡更像表单页的一部分。
- 预览轮播不可抢占表单的视觉重心。

**理由**：

- 当前登录页结构已正确，问题在于还偏组件化，没有完全贴近 PDF。
- 继续强化 preview 只会把登录页重新推向“匿名首页”。

**替代方案**：把预览区做得更抢眼，以提高“内容感”。

- 被否决原因：会削弱登录页的主任务。

### 5. 详情页改为“头卡轻、正文稳、动作局部化”

**选择**：详情页保留现有结构，但降低首卡与正文卡的容器感，强化标题、时间、原文按钮、正文的稳定层级。

**具体方向：**

- 白底 secondary header，图标和局部动作更轻。
- 首卡左右边距收窄，标题区和时间区更紧凑。
- 原文按钮不再过满、过粗。
- 正文承载优先，卡片只是内容边界，而不是页面主角。

**理由**：

- 当前详情页虽然可用，但整体仍偏“大白卡承载所有内容”的通用实现。
- PDF 的详情页更像“内容页”，而不是“系统卡片页”。

**替代方案**：保留现有详情结构，只调按钮和字号。

- 被否决原因：不足以改变详情页整体气质。

### 6. 视觉验收采用“截图对稿 + 负面清单”双轨

**选择**：本 change 的验收不只看功能和编译，还要明确视觉验收规则。

**验收包括：**

- 首页、登录页、频道页、详情页各自提供真机或模拟器截图。
- 对照 PDF 核对标题层级、顶部栏样式、控制区厚度、卡片密度、留白节奏。
- 同时用负面清单排查：
  - 是否仍存在过强 hero
  - 是否仍有多层卡片嵌套
  - 是否仍有与 PDF 不一致的整条蓝色频道顶栏
  - 是否存在过厚按钮、过重阴影、过散间距

**理由**：

- 视觉问题无法只靠代码 review 判断。
- 引入负面清单能避免“看起来也还行”的泛化 UI 混过验收。

## Risks / Trade-offs

- [Risk] 追求对稿可能导致页面过于保守 → Mitigation: 重点收敛气质和层级，不要求逐像素复刻。
- [Risk] 首页与频道页同时调整，可能引发公共组件连锁改动 → Mitigation: 先定义页面家族，再局部重构共享组件。
- [Risk] 过度削弱装饰后页面可能显得平淡 → Mitigation: 保留蓝白主色和少量品牌强调，但让强调只出现在关键层级。
- [Risk] PDF 截图与现有真实数据长度差异会影响观感 → Mitigation: 将验收重点放在容器、层级、节奏和密度，而不是固定文案长度。

## Migration Plan

1. 先扩展并收敛视觉 token，明确颜色、圆角、阴影、边框、间距和 header 家族规则。
2. 优先调整首页入口页、频道页顶部和控制区，解决最大偏差来源。
3. 再收紧列表卡片与详情页结构，统一内容区节奏。
4. 最后精修登录页和空态、加载态，补齐截图验收。

## Open Questions

- 首页入口页是否完全去掉大副标题，只保留品牌名和频道入口。
- 频道页页面标题是否保留“返回箭头 + 页面标题”双元素，还是进一步压缩成更轻的单线头部。
- 详情页正文表格区域是否需要在视觉上做额外去边框处理，还是保留站点原始表格边界作为正文的一部分。
