## Context

当前企业认证相关能力是分散拼起来的：`auth-register` 仍按较重的字段集建模，`auth-audit-status` 在“无申请记录”时返回 404，前端页面则分别在注册页、审核状态页、我的页里各自处理这些差异。结果是用户虽然能提交认证，但体验上仍然偏重，后端契约也不够统一。

这个项目的小程序并不需要一套复杂的企业入驻系统，更现实的方向是“登录后，用最少字段完成可信认证，后续状态一目了然”。约束条件也很明确：
- 当前已有手机号验证码登录，可把企业认证绑定到登录账号
- 当前没有成熟的第三方 OCR / 企业信息核验服务接入
- 前端已经有注册页、审核状态页、我的页，适合在现有页面族内重构，而不是再开新流程
- 审核台与采集运维后台将由独立 admin change 负责，本 change 只定义用户侧体验与其依赖的认证契约

## Goals / Non-Goals

**Goals:**
- 将企业认证定义为已登录用户的一步式认证动作，而不是独立匿名注册流程
- 将提交字段精简到“足够审核”的最小集合，减少重复输入
- 统一认证状态契约，让前端不再依赖 404 来判断“未提交认证”
- 明确 pending / approved / rejected / none 四类状态下的下一步动作
- 允许被驳回用户在上次资料基础上重提，避免重复填写

**Non-Goals:**
- 不引入第三方企业数据库核验或付费实名服务
- 不在本次变更中实现 OCR 识别营业执照；如果未来接入，只作为增强，不作为流程前置
- 不设计后台审核工作台；本次只约束用户侧契约与状态模型
- 不承担采集任务管理、企业后台管理等运营端后台能力
- 不改动手机号登录方式本身

## Decisions

### 1. 企业认证绑定登录账号
- **Decision:** `POST /api/auth/register` 改为要求有效 Bearer token；认证申请始终归属于当前登录用户。
- **Why:** 这样可以直接复用登录手机号、避免匿名注册与用户体系脱节，也能让“我的”页自然成为认证入口。
- **Alternative considered:** 保持匿名提交。放弃，因为会造成一个手机号登录账号对应多份不明来源申请，状态同步也更乱。

### 2. 提交字段缩到最小可审核集合
- **Decision:** 必填字段保留 `companyName`、`creditCode`、`licenseImage`；`contactPhone` 改为可选，默认取当前登录手机号；`contactName` 改为可选。
- **Why:** 对人工审核来说，营业执照图片 + 企业名称 + 信用代码已经是最核心的核验依据；联系方式只需要能联系到当前账号即可。
- **Alternative considered:** 保留联系人姓名和联系电话为强必填。放弃，因为这类字段对审核有效性帮助有限，却会显著增加填写负担。

### 3. 审核状态接口统一返回状态机，不再用 404 表示“未提交”
- **Decision:** `GET /api/auth/audit-status` 统一返回 `status: none|pending|approved|rejected`，同时附带 `nextAction`。
- **Why:** 前端最适合消费的是状态机，而不是 HTTP 语义混搭。这样 profile、register、audit-status 三页可以共用一套判断分支。
- **Alternative considered:** 继续用 404 表示无记录。放弃，因为这会把“正常的未认证状态”误建模成异常分支。

### 4. 用户只维护一条当前有效申请
- **Decision:** 同一用户在任意时刻只存在一条“当前有效的企业认证申请视图”。pending 时不允许重复提交；rejected 时在最近一次申请基础上重提；approved 时默认只读。
- **Why:** 这样最符合小程序场景，也能让审核状态和表单预填都简单很多。
- **Alternative considered:** 每次提交都插入全新申请。放弃，因为会让状态页、profile 页和人工审核都面对多条记录选择问题。

### 5. OCR 作为未来增强，而不是当前前置能力
- **Decision:** 这次只要求上传营业执照并允许手动填写/修正，不把 OCR 识别作为依赖。
- **Why:** 这是当前最简单、最稳的落地方案；如果以后接 OCR，可以只做“自动填充建议”，不改变主流程。
- **Alternative considered:** 把“上传执照自动识别”作为本次必须能力。放弃，因为会引入外部依赖和失败兜底复杂度。

## Risks / Trade-offs

- [Risk] 精简字段后，审核人员可用的信息更少 → Mitigation: 保留营业执照图片、企业名称、信用代码和登录手机号，保证最低可核验信息集
- [Risk] 单申请模型会让历史提交记录处理变复杂 → Mitigation: 用户侧只暴露“当前有效申请”，历史记录可留在数据库内部字段或审计表中，前端不感知
- [Risk] 审核状态统一返回 200 后，前端可能误把鉴权失败也当成普通状态 → Mitigation: 未登录仍保持 `code: 401` 路径，只有已登录用户才进入 `none|pending|approved|rejected` 状态机
- [Risk] 没有 OCR 时，用户仍需手填企业信息 → Mitigation: 先通过精简字段和登录手机号复用降低成本，OCR 留作后续增强点

## Migration Plan

1. 先修改认证契约和主 specs，统一字段与状态机语义。
2. 后端调整 `auth-register` 和 `auth-audit-status` 的请求/响应模型，并处理单用户单有效申请策略。
3. 前端注册页改为最小字段表单，并支持基于审核状态的预填/禁用策略。
4. 前端 profile / audit-status 页切换到统一状态机，移除对 404 无记录分支的依赖。
5. 与独立后台审核 change 对齐审核结果的产出格式，再联调验证未登录、未认证、审核中、通过、驳回重提五类路径。

## Open Questions

- 审核通过后，是否允许用户主动发起“变更企业信息”申请，还是先完全只读？
- 驳回后重提是覆盖上一条申请，还是保留历史并生成新版本号？用户侧建议只看到最新版本，但后端是否需要保留版本历史仍待定。
