## Why

当前 `ui` 项目混合了承载管理后台的 `/admin/*` 页面与一套面向用户的 Web 小程序样例页面，但项目的正式用户端已经是 `miniapp`。继续保留这套样例代码会让系统边界模糊、需求判断失真，并增加后续维护和误改成本，因此需要把 `ui` 明确退役并从仓库中移除。

## What Changes

- **BREAKING** 删除 `ui` 项目中的前端样例代码，不再把它视为正式交付或继续维护的前端工程。
- **BREAKING** 移除 `ui` 中当前承载的 `/admin/*` 页面代码，因为该项目整体被定义为样例产物而非正式后台实现。
- 明确项目正式系统边界：用户端以 `miniapp` 为准，后端服务以 `server` 为准，采集链路以 `crawler` 为准；`ui` 不再作为正式系统组成部分。
- 清理仓库内对 `ui` 作为正式系统的定位描述，避免后续文档、需求和实现继续把 `ui` 计入交付范围。
- 为后续如果需要真正的后台管理平台，保留“重新立项/重新提案”的空间，而不是沿用样例工程继续演进。

## Capabilities

### New Capabilities
- `ui-sample-retirement`: 定义如何识别、下线并从仓库中移除 `ui` 样例工程，同时更新系统边界说明

### Modified Capabilities
- `ui-api-binding`: 取消当前 `ui` 工程作为正式前端的定位，移除其与后端 API 绑定的要求
- `admin-crawl-run-control`: 取消当前由 `ui` 的 `/admin/crawl` 页面承载的控制台前端，明确服务端接口保留但样例前端移除

## Impact

- 影响代码：`ui/` 全目录及任何直接引用 `ui` 作为正式前端的说明
- 影响文档：项目架构、需求清单、联调说明中关于 `ui` 是正式管理后台的表述
- 影响交付边界：正式用户端只保留 `miniapp`，正式服务端保留 `server`，正式采集链路保留 `crawler`
- 影响现有后台可视入口：`/admin/reviews`、`/admin/companies`、`/admin/crawl` 的样例 Web 前端会消失，但对应服务端接口可以继续存在
