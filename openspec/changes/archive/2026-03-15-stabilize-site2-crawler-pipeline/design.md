## Context

site2 当前已经具备签名、验证码识别、代理会话、列表抓取、详情抓取和 SQLite 落库能力，但整体仍停留在试跑阶段：历史测试数据混入正式表、`59` 与 `00101` 的覆盖不完整、`incremental` 与 `recovery` 任务接口不一致、失败重试与补偿逻辑不足、缺少源站对账能力。目标不是单次抓取成功，而是把 site2 变成一个可长期运行的数据管道：支持从 `2026-03-01` 到当前时间的正式初始化，支持后续每 2 小时稳定增量，支持手动重复执行而不产生重复脏数据，并在异常情况下通过补偿和对账尽量消除遗漏。

现有 `notices` 表已经以 `(site, id)` 为主键并使用 upsert，这为幂等提供了基础。设计重点在于把 site2 的任务执行模型、窗口策略、失败处理和验收标准统一起来，让“重复运行可收敛”和“漏抓可发现、可补回”成为显式系统行为。

## Goals / Non-Goals

**Goals:**
- 使 site2 的 `backfill`、`incremental`、`recovery` 共享同一套窗口抓取与落库内核，避免链路分叉
- 定义正式初始化流程：清理 `site2_ccgp_sichuan` 历史测试数据后，回填 `2026-03-01` 到当前时间的 `59` 与 `00101`
- 强化幂等性：同一窗口重复抓取、手动补跑、定时重复执行都只会收敛到单条 `(site, id)` 记录，并允许新运行修正旧记录字段
- 提升防遗漏能力：通过安全回扫窗口、最近 48 小时补偿和日级对账来降低验证码、代理、分页空返回带来的漏抓风险
- 定义稳定性验证方式，包括源站 total 对账、重复执行一致性、失败恢复和字段完整性检查

**Non-Goals:**
- 不改变 `notices` 表 schema 或站点二 API 协议
- 不引入新的外部存储、消息队列或分布式调度系统
- 不保证在单次任务执行中绝对零失败；目标是通过幂等重跑和补偿实现最终收敛
- 不处理 site1 的采集策略重构，除非为了抽取共享工具所必需

## Decisions

### 1. 以共享的窗口执行内核统一 backfill / incremental / recovery

**选择**：将 site2 三类任务统一到同一套 `process_window` / `run_window_series` 风格的执行内核，所有入口都显式传入数据库连接、窗口序列和类型集合。

**理由**：当前 `incremental` 与 `recovery` 的调用签名已经与 `process_window` 脱节，说明这三条链路没有被当作一个系统维护。共享执行内核可以让分页、会话重建、详情补抓、落库和日志逻辑只维护一处，减少“backfill 能跑、增量不能跑”的漂移。

**替代方案**：分别修复三类任务，但保留各自独立实现。这样改动小，但后续最容易再次分叉。

### 2. 使用“重叠增量 + 48h recovery”替代“严格无重叠增量”

**选择**：每 2 小时运行的增量任务不只抓上一个精确 2 小时窗口，而是回扫一个带重叠的安全窗口；同时保留每天一次最近 48 小时补偿任务。

**理由**：site2 存在验证码、代理和源站发布时间延迟等不稳定因素。严格无重叠窗口在边界时刻最容易漏数据；有主键 upsert 的前提下，多抓并去重远比少抓漏数据安全。48 小时补偿则负责修复单次增量失败、详情暂时失败和源站延迟可见的问题。

**替代方案**：只保留精确 2 小时窗口，不做重叠和 recovery。实现简单，但无法提供“最终补齐”的稳定性保证。

### 3. 将“清理历史测试数据 + 正式 backfill”视为一次迁移，而不是就地修补

**选择**：对 `site='site2_ccgp_sichuan'` 的历史测试数据执行一次显式清理，再做正式初始化 backfill。

**理由**：当前库中混有 `category_num = NULL`、覆盖不完整和日期范围不一致的数据，继续在其上打补丁会让后续对账和验收失真。清理后重建基线，更容易判断回填是否完整，也能避免把测试痕迹误当生产数据。

**替代方案**：直接重跑并依赖 upsert 修正旧数据。虽然技术上可行，但无法保证旧测试数据一定被完全覆盖，也会增加验收难度。

### 4. 用“源站 total 对账 + DB count 对账”定义稳定性验收

**选择**：引入显式对账输出，按日期、按类型比较源站 `probe_total` 与数据库记录数；必要时进一步按窗口输出差异。

**理由**：没有对账就无法判断是否遗漏。site2 的稳定性不能只靠“任务返回成功”判断，而必须有一个可重复执行的验收视角。日级对账是最直接、成本最低且可自动化的标准。

**替代方案**：只看日志中的 saved 数量。日志能反映执行过程，但不能证明最终覆盖完整。

### 5. 允许“不完整记录先入库，但必须可被后续运行修正”

**选择**：保持 `(site, id)` upsert 模式，允许在详情临时失败时先写入基础字段，但要求后续重跑、补偿或 recovery 能覆盖为更完整的最终状态；同时增加字段完整性验证，避免长期残留大批空字段记录。

**理由**：完全阻塞式地等待所有详情成功，容易把短时波动放大成整体失败。先保住主记录，再通过补偿填全字段，更符合幂等管道的思路。

**替代方案**：详情失败则整条不入库。数据更干净，但在代理/验证码不稳定时更容易产生不可见漏抓。

## Risks / Trade-offs

- **重叠增量会增加重复请求成本** -> 依赖 `(site, id)` upsert 去重，并限制重叠窗口范围在可接受的 4-6 小时级别
- **清理历史数据后正式回填耗时较长** -> 分类型、分日执行，并用日级对账逐步确认完成情况
- **详情失败后先写基础字段可能造成短暂不完整记录** -> 用 recovery 和字段完整性检查保证后续补齐，并把长期空字段视为失败信号
- **源站 total 可能随时间略有波动** -> 对账以固定回填时点为准，并优先关注明显缺口而非单条抖动
- **代理/验证码不稳定仍可能导致单次任务失败** -> 通过会话重建、重叠回扫和 48h recovery 将失败从“永久遗漏”降为“延迟补齐”

## Migration Plan

1. 在实现前明确 site2 正式运行参数：初始化起点、增量安全回扫范围、recovery 频率与对账口径
2. 修正 site2 三类任务，统一到共享执行内核，并补齐日志与返回结果
3. 在非正式数据基线上做幂等与对账测试，确认重复执行不会新增重复记录
4. 清理 `site2_ccgp_sichuan` 历史测试数据
5. 运行 `2026-03-01` 到当前时间的正式 backfill，按日/按类型持续对账
6. 开启每 2 小时增量与 48h recovery，并持续观察对账结果

回滚策略：若正式 backfill 或新任务链路出现系统性异常，可暂停 site2 定时运行并重新回到“仅手工验证”状态；由于 site2 数据按 site 隔离，必要时可以再次清理 `site2_ccgp_sichuan` 数据后重建。

## Open Questions

- 增量安全回扫窗口最终定为最近 4 小时、6 小时，还是“上一个完整 2 小时 + 前一个窗口”
- recovery 是每日一次即可，还是在早期稳定阶段需要更高频率
- 对账输出是只写日志，还是需要落成独立报表/脚本输出，供人工核验
