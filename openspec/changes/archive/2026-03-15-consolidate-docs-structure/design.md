## Context

当前 `docs/` 目录同时承载了多个层次的内容：项目总览、前后台接口、数据库设计、数据源契约、联调说明，以及抓包原始数据、给外部模型的提示词等过程材料。这些文档并不是同一类型，但目前全部平铺在根目录，导致两个直接问题：

1. 正式文档入口不清晰，阅读者难以判断哪份才是当前有效的主文档。
2. 过程材料和正式文档混在一起，后续维护时容易继续叠加重复或历史遗留内容。

这次 change 的重点不是新增业务能力，而是把文档体系从“自然生长”收敛成“有主入口、有归档边界、有主题合并策略”的结构。

## Goals / Non-Goals

**Goals:**
- 明确 `docs/` 根目录只保留正式长期维护的主文档
- 定义哪些文档需要合并成统一入口，尤其是后台管理相关文档
- 定义哪些材料属于过程性或历史性内容，应迁出根目录
- 减少数据库设计和后台接口说明的重复表达
- 为后续继续写文档提供稳定的分类规则

**Non-Goals:**
- 不修改运行时代码或 API 行为
- 不在本 change 中重写所有文档内容，只做结构性收敛和主从关系调整
- 不要求一次性补齐所有缺失接口文档，只要求确定统一入口和缺口标记方式
- 不把 `docs/` 改造成复杂文档站点，只做仓库内 Markdown 目录收敛

## Decisions

### 1. `docs/` 根目录只保留正式主文档
- **Decision:** `docs/` 根目录只保留项目总览、前台接口、后台接口、后台需求、数据源接口、数据库设计、联调指南等正式文档。
- **Why:** 根目录应该承担“第一次进入项目时能快速找到正式入口”的职责，而不是承载所有研究过程。
- **Alternative considered:** 保持所有 Markdown 平铺在根目录。放弃，因为当前已经出现正式文档与过程材料混杂的问题。

### 2. 后台文档合并为统一后台接口入口
- **Decision:** 将管理员审核接口文档与 crawl control plane 说明收敛成统一的后台管理接口文档。
- **Why:** 统一运营后台已经成为正式系统方向，后台接口不应再拆成“审核一份、采集一份、控制面说明一份”的松散结构。
- **Alternative considered:** 保留分散文档，并在 PRD 中交叉引用。放弃，因为后续后台前端实现需要一份直接可查的统一接口入口。

### 3. 数据库设计确定主文档与补充说明
- **Decision:** `数据库表设计.md` 作为数据库设计主文档，`存储表结构说明.md` 要么并入主文档，要么降级为补充说明，不再与主文档并列承担权威角色。
- **Why:** 当前两份文档在 notices 表、字段来源和落库约定上有明显交叉，继续并列维护会导致漂移。
- **Alternative considered:** 两份都保留且长期并列。放弃，因为这会让“到底看哪份”为长期问题。

### 4. 过程性材料迁出正式文档根目录
- **Decision:** 抓包原始数据、辅助提示词等内容迁出 `docs/` 根目录，进入 `docs/archive/` 或其他更适合的归档位置。
- **Why:** 这类内容有价值，但价值在于追溯和研究，不在于作为正式文档入口。
- **Alternative considered:** 直接删除所有过程材料。暂不默认采用，因为部分材料仍有回溯价值。

### 5. 缺失接口文档允许保留在统一入口中并显式标记
- **Decision:** 对于后台登录、Dashboard 聚合接口等尚未实现的接口，可以先写入统一后台接口文档，并明确标记“未完成”。
- **Why:** 这样可以把“已有接口”和“明确缺口”放在同一个入口里，减少后续再次分散记录。
- **Alternative considered:** 只有已实现接口才能进入正式文档。放弃，因为这会让需求缺口失去统一记录位置。

## Risks / Trade-offs

- [Risk] 合并文档后，部分旧链接或旧引用会失效
  Mitigation: 在迁移时保留明确的新旧对应关系，并在必要时添加说明

- [Risk] 归档材料迁走后，团队成员误以为这些材料被删除
  Mitigation: 在 design 和 tasks 中明确是“迁出根目录”而不是“一律删除”

- [Risk] 后台接口文档统一后，仍可能混入需求性描述
  Mitigation: 保持“需求清单”和“接口文档”分层，前者描述页面和能力，后者描述接口契约与完成状态

- [Risk] 数据库文档合并时可能误删仍有参考价值的字段说明
  Mitigation: 先明确主从关系，再决定是全文并入还是仅保留补充章节

## Migration Plan

1. 先确定正式文档名单与目标目录结构。
2. 新建统一后台接口文档，并把后台接口说明迁移进去。
3. 决定数据库文档的主从关系，并做收敛。
4. 将抓包数据、提示词等过程材料迁出根目录到归档位置。
5. 更新总览类文档中的文档引用，确保入口一致。

回滚策略：
- 若某次合并导致信息丢失或引用混乱，可恢复原文件并重新拆分；
- 由于这是文档整理 change，不影响运行时代码，回滚成本较低。

## Open Questions

- 抓包原始数据最终应进入 `docs/archive/`，还是放到更贴近爬虫目录的研究目录中？
- `存储表结构说明.md` 更适合完全并入数据库主文档，还是保留为 notices 单表的补充说明？
- 统一后台接口文档是否在本次整理中同时补齐“未完成接口”章节，还是只先做已实现接口合并？
