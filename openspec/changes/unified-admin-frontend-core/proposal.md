## Why

当前仓库已经具备一部分管理员能力的服务端接口，但没有一个边界清晰、信息架构稳定的正式运营后台前端。现在需要先把“统一运营后台最核心必须具备哪些功能、页面和模块”定义清楚，这样后续即使部分 API 还未补齐，也能按统一蓝图推进后台建设，而不是边做边猜。

## What Changes

- 新增统一运营后台前端的核心能力规划，明确后台第一阶段必须具备的模块、页面和导航结构。
- 定义后台基础壳层要求，包括登录入口、布局、导航、全局状态、环境标识和权限守卫。
- 定义企业审核与企业目录作为后台的核心业务模块，明确哪些页面和交互属于第一优先级。
- 定义采集控制与运行记录作为后台的核心运维模块，明确必须具备的动作、历史、详情和异常反馈视图。
- 定义一个最小但可扩展的运营总览页，用于承载数据新鲜度、待处理事项和异常提醒。
- 明确哪些功能可以先以占位态、只读态或“等待 API”状态落地，而不是因为接口未齐全就推迟整个后台规划。

## Capabilities

### New Capabilities
- `admin-frontend-shell`: 统一运营后台的基础壳层、导航、权限入口和全局页面框架
- `admin-operations-dashboard`: 统一运营后台首页的总览、提醒和关键状态视图
- `admin-enterprise-operations`: 企业审核中心和企业目录的核心页面与交互要求
- `admin-crawl-operations-console`: 采集控制、运行历史和运行详情的前端控制台要求

## Impact

- 影响产品规划：统一运营后台将从零开始作为正式系统规划，而不是继续沿用旧样例前端
- 影响前端工程结构：后续需要在当前仓库内新增一个明确命名的后台前端工程，例如 `admin-frontend/`
- 影响服务端协作：现有 `/api/admin/*` 将成为后台第一阶段的重要数据来源，同时会暴露缺失 API 的补齐需求
- 影响信息架构：管理员能力将从“若干散落接口”收敛成企业审核、企业目录、采集控制、运行记录、总览五类核心体验
