## Context

当前后台企业审核与企业目录在实现上高度重复：两者共用同一套列表列定义，详情页也基本复用同一份申请数据，只是时间字段和入口路径略有不同。同时，管理员身份仍是单一固定账号，后端虽然有 `audited_by` 字段，但实际无法区分不同审核员，导致审核留痕接近失效。

这次变更不是单纯增加几个按钮，而是要同时理顺三层对象：

- 审核员身份：谁在操作
- 企业认证申请：一次提交、一次审核
- 企业档案：企业当前生效状态

如果继续沿用单一 admin 身份并在两个重复页面上叠加按钮，后续“编辑企业”“删除测试数据”“谁审核的”都会越来越混乱。

## Goals / Non-Goals

**Goals:**

- 在第一阶段建立轻量但真实可用的后台身份模型：1 个超级管理员 + 2 个审核员
- 让审核动作写入真实审核人，并在列表/详情中可见
- 将审核队列和企业目录收敛为同一个“企业管理”模块，在同模块内区分申请视图和企业档案视图
- 明确高风险动作边界，让删除测试数据、编辑企业档案、作废申请仅由超级管理员执行

**Non-Goals:**

- 不引入完整 RBAC、组织架构、权限配置后台
- 不在本阶段支持任意数量后台账号自助管理
- 不重构小程序端企业认证流程
- 不在本阶段补齐完整操作日志中心，只先保证审核人与关键动作留痕

## Decisions

### 1. 采用“两类角色、三个固定账号”的轻量后台身份模型

第一阶段不做完整后台用户表与权限系统，而是支持：

- `super_admin`：1 个，保留当前 admin 的最高权限定位
- `reviewer`：2 个，用于日常审核

登录态仍通过后台 JWT 承载，但 token 中必须带 `adminId`、`username`、`role`。这样能以最低复杂度解决“谁审核的”问题，并给高风险动作加角色边界。

备选方案：

- 保持单一 admin，仅在审核操作时手工选择审核人。未采用，因为这是假审计，责任无法闭环。
- 直接上完整后台账号体系。未采用，因为对当前项目阶段过重，且实现成本远大于业务收益。

### 2. 将“审核队列”和“企业目录”收敛为同一模块，但保留两种业务视图

模块层面只保留一个“企业管理”，页面内通过 tabs / 视图切换区分：

- 申请视图：待审核、已处理、全部申请
- 企业档案视图：当前企业档案

这样做可以复用筛选和表格骨架，减少重复页面；同时又不会把“申请单”和“企业档案”混成同一对象。

备选方案：

- 继续保留两个侧边菜单。未采用，因为两页重复度过高，后续每次改字段或筛选都要双改。
- 完全合并成一个无区分的大列表。未采用，因为申请工作流和企业主数据的操作语义不同，详情页会变得含混。

### 3. 将动作边界绑定到对象语义，而不是页面位置

本变更中动作边界按对象划分：

- 申请单：通过、驳回、作废
- 企业档案：查看、编辑、删除测试数据

权限再叠加在动作上：

- 审核员：通过、驳回、查看
- 超级管理员：额外拥有作废申请、编辑档案、删除测试数据

这样能避免“审核页里既改申请又改企业”的语义混乱。

### 4. 测试数据删除仅作为超管受控能力

需求里已经明确存在“测试数据无法删除”的痛点，因此第一阶段必须给出路径。但删除动作只对超级管理员开放，并优先限定为测试数据或显式确认后的危险操作。

备选方案：

- 所有人都可删除。未采用，因为审核员不应拥有破坏性主数据操作。
- 完全不提供删除，只允许作废。未采用，因为现阶段确有测试数据清理需求，纯作废会长期污染企业目录。

## Risks / Trade-offs

- [固定账号数量过少，后续仍需扩展] → 本阶段先把 token 结构和角色模型做对，后续再平滑扩展到后台用户表
- [申请与企业档案关系定义不清] → 明确“申请单是历史记录，企业档案是当前有效状态”，详情页分别承载不同动作
- [删除测试数据误伤正式数据] → 将删除限制为超级管理员，并要求二次确认或显式测试标识/受控入口
- [UI 合并后信息过载] → 只合并模块与列表壳子，不把申请详情和档案详情强行做成一个页面

## Migration Plan

1. 扩展后台登录与鉴权 payload，使 `role/adminId/username` 进入管理员 token
2. 更新审核接口，在 approve/reject 时写入真实审核人
3. 在列表与详情中展示审核人和审核时间
4. 将侧边栏“审核队列 / 企业管理”收敛为一个企业管理模块，并增加申请视图/档案视图切换
5. 为超级管理员增加受控的测试数据删除与企业档案编辑入口
6. 用现有历史申请数据验证：旧数据允许 `auditedBy` 为空，新审核记录必须带真实审核人

## Open Questions

- 测试数据删除是物理删除，还是先做“作废/隐藏 + 超管删除”双层策略
- 企业档案编辑是否直接改最新申请记录，还是单独引入企业档案写模型
