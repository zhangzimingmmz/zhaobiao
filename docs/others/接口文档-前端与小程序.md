# 金堂招讯通 — 前端与小程序接口文档

本文档面向**小程序与 Web 前端**，描述其调用的**自有后端 API** 的契约（Method、Path、请求参数/体、响应结构、字段定义）。  
**与《原始数据接口文档》的区分**：《原始数据接口文档》描述的是**爬虫与数据源**（网站一、网站二）之间的契约；本文档描述的是**前端/小程序与自有后端**之间的接口，数据由后端从爬虫落库或聚合后提供给前端，前端不直接调用数据源。

---

## 文档约定

- **Base URL**：前端与小程序通过配置注入，文档中以占位符 `{{baseUrl}}` 表示，例如开发环境 `https://api-dev.example.com`、生产环境 `https://api.example.com`。实际请求 URL = `{{baseUrl}}` + Path。
- **通用请求头**：
  - `Content-Type`：请求体为 JSON 时使用 `application/json`。
  - 鉴权：已登录用户请求时携带 `Authorization: Bearer <token>`（或后端约定的其他头/参数）；具体 token 获取方式见「登录」接口。未登录可访问的接口（如列表、详情、获取验证码）可不带鉴权。
- **响应码与错误体**：
  - HTTP 200 表示请求被接受；业务成功与否看响应体中的 `code` 或 `success`。约定：`code === 200` 或 `success === true` 为成功；否则为业务失败，前端根据 `message` 或 `msg` 展示错误信息。
  - 错误响应体示例：`{ "code": 400, "message": "参数错误" }` 或 `{ "success": false, "msg": "验证码已过期" }`。具体字段名以后端实现为准，文档中统一用「错误信息」字段。
- **分页与筛选**：
  - 分页：统一使用 `page`（页码，从 1 开始）与 `pageSize`（每页条数）。响应中提供 `total`（总条数）与当前页数据列表。
  - 筛选参数命名见各接口的「请求参数」表；与首页一级/二级菜单及筛选区一致。
- **日期时间展示**（小程序）：
  - 后端返回的 `publishTime`、`expireTime`、`auditTime`、`createdAt` 等可能为 ISO 或原始字符串。
  - 小程序统一使用 `miniapp/src/utils/formatDate.js`：`formatDate` 输出 `YYYY-MM-DD`（列表、卡片、发布时间）；`formatDateTime` 输出 `YYYY-MM-DD HH:mm`（详情、审核、提交时间）。

## 接口清单

| 接口用途       | Method | Path（相对 {{baseUrl}}）     | 说明 |

## 一、列表接口

### 1.1 契约摘要

| 项     | 值 |
|--------|-----|
| 接口标识 | list |
| Method | GET |
| Path   | `{{baseUrl}}/api/list` |

### 1.2 请求参数（Query）

| 参数         | 类型   | 必填 | 说明 |
|--------------|--------|------|------|
| page         | number | 是   | 页码，从 1 开始 |
| pageSize     | number | 是   | 每页条数，建议 10～20 |
| category     | string | 是   | 业务类型，与首页一级/二级菜单对应，见下表 |
| keyword      | string | 否   | 搜索关键词（对应「搜索」筛选项） |
| timeStart    | string | 否   | 发布时间起，如 `2026-03-01 00:00:00` |
| timeEnd      | string | 否   | 发布时间止 |
| regionCode   | string | 否   | 地区代码（四川 21 地市等），对应「地区/区划」筛选 |
| source       | string | 否   | 交易来源/平台（如省级/市级/区县），对应「交易来源」 |
| purchaseManner | string | 否   | 采购方式（政府采购-采购公告时使用），如 1=公开招标 |
| purchaseNature | string | 否   | 采购性质（政府采购-采购公告时使用），1=货物、2=工程、3=服务 |
| purchaser    | string | 否   | 采购人关键词（政府采购-采购意向公开时使用） |

**category 取值与首页菜单对应**：

| 首页一级菜单 | 二级菜单       | category 建议取值 |
|--------------|----------------|-------------------|
| 工程建设     | 工程建设       | `002001009`（招标计划）或 `002001001`（招标公告），可由前端传其一或由后端约定合并 |
| 工程建设     | 政府采购       | `002002001`（政府采购采购公告，网站一） |
| 政府采购     | 采购意向公开   | `59`（网站二 noticeType） |
| 政府采购     | 采购公告       | `00101`（网站二 noticeType） |
| 信息展示     | （无二级）     | 由后端约定取值，如 `info`；仅 keyword 生效 |

说明：后端可根据 category 映射到《原始数据接口文档》中的 site1-list condition（002001009/002001001/002002001）或 site2 的 noticeType（59/00101），并聚合为统一列表响应。

**筛选说明**：当 category=002002001（政府采购采购公告）时，数据源（网站一）的 tradingsourcevalue 多为空，source 筛选可能无结果；建议前端在无结果时提示用户可尝试其他筛选条件。

### 1.3 响应结构

- 顶层：`code`（或 `success`）、`data`（对象）。`data` 包含：
  - `total`：number，总条数。
  - `list`：数组，当前页列表项。

### 1.4 列表单条字段表

以下字段为前端列表展示与跳转详情所需；字段命名与《原始数据接口文档》中落库字段对齐，便于后端透传或简单映射。

| 字段          | 类型   | 可空 | 说明 | 对应原始数据 |
|---------------|--------|------|------|----------------|
| id            | string | 否   | 业务唯一键，详情请求用 | site1-list records.id；site2 data.rows.id |
| title         | string | 否   | 标题 | site1 title；site2 title |
| publishTime   | string | 可空 | 发布时间，如 `2026-03-14 11:14:25` | site1 webdate；site2 noticeTime |
| sourceName    | string | 可空 | 来源/发布单位名称 | site1 zhuanzai；site2 author |
| regionName    | string | 可空 | 地区名称 | site2 regionName；site1 可由 tradingsourcevalue 映射 |
| regionCode    | string | 可空 | 地区代码 | site2 regionCode；site1 tradingsourcevalue |
| categoryNum   | string | 可空 | 分类/公告类型编号 | site1 categorynum；site2 noticeType |
| categoryName  | string | 可空 | 分类/公告类型名称（展示用） | 可由后端根据 categoryNum 映射 |
| originUrl     | string | 可空 | 原文链接（详情页「查看原文」用） | site1 Base+linkurl；site2 详情 originUrl 或由后端拼装 |
| summary       | string | 可空 | 摘要（列表卡片展示） | site1 content；site2 description |
| planId        | string | 可空 | 网站二采购公告详情用，可选 | site2 rows.planId |
| purchaseNature | string | 可空 | 采购性质，1=货物、2=工程、3=服务 | site2 purchaseNature；site1 无此字段 |

## 四、认证与用户接口

### 4.1 获取验证码

| 项     | 值 |
|--------|-----|
| Method | GET |
| Path   | `{{baseUrl}}/api/auth/captcha` |

- **请求**：无 body；可选 query `mobile`（手机号），用于与登录请求关联（若后端按手机号限流）。
- **响应**：成功时返回验证码图片。可选形式：① 响应体为图片二进制，Content-Type `image/jpeg`；② 或 JSON `{ "code": 200, "data": { "imageBase64": "..." } }`。  
- **Mock 约定**：若后端未实现，前端可 Mock 为固定图片或固定验证码（如 `123456`）用于联调。

### 4.2 登录

| 项     | 值 |
|--------|-----|
| Method | POST |
| Path   | `{{baseUrl}}/api/auth/login` |

- **请求体**：`{ "mobile": "11位手机号", "captcha": "验证码" }`。
- **成功响应**：`{ "code": 200, "data": { "token": "...", "userId": "...", "mobile": "..." } }`。前端将 token 存入本地并在此后请求头中携带 `Authorization: Bearer <token>`。
- **失败响应**：`{ "code": 400, "message": "验证码错误" }` 等，前端展示 message。

**Mock 约定**：若后端未实现，可约定任意验证码均返回固定 token，便于前端开发。

### 4.3 企业认证提交

| 项     | 值 |
|--------|-----|
| Method | POST |
| Path   | `{{baseUrl}}/api/auth/register` |

- **鉴权要求**：需携带有效 Bearer token（已登录用户）
- **请求体**：

| 字段         | 类型   | 必填 | 说明 |
|--------------|--------|------|------|
| companyName  | string | 是   | 企业名称 |
| creditCode   | string | 是   | 统一社会信用代码，18 位 |
| contactName  | string | 否   | 联系人姓名（可选） |
| contactPhone | string | 否   | 联系电话，11 位（可选，默认使用登录手机号） |
| licenseImage | string | 是   | 营业执照图片 URL 或 base64（由后端约定上传接口或直传） |

- **成功响应**：`{ "code": 200, "data": { "applicationId": "...", "status": "pending", "companyName": "...", "creditCode": "...", "contactPhone": "..." } }`，前端跳转审核状态页。
- **失败响应**：
  - 未登录：`{ "code": 401, "message": "请先登录" }`
  - 参数错误：`{ "code": 400, "message": "参数错误：companyName 不能为空" }`
  - 审核中重复提交：`{ "code": 409, "message": "已有审核中的认证申请", "data": { "status": "pending" } }`
  - 已认证用户重复提交：`{ "code": 409, "message": "企业认证已通过，如需变更请联系管理员" }`

- **重提机制**：
  - 当前用户状态为 `rejected` 时，再次提交会更新现有申请并重置状态为 `pending`
  - 当前用户状态为 `pending` 时，不允许重复提交
  - 当前用户状态为 `approved` 时，不允许重复提交

### 4.4 审核状态查询

| 项     | 值 |
|--------|-----|
| Method | GET |
| Path   | `{{baseUrl}}/api/auth/audit-status` |

- **请求**：需鉴权（Bearer token）。
- **响应**：统一返回状态机快照，包含 `status` 和 `nextAction` 字段。

**响应结构**：`{ "code": 200, "data": { ... } }`

**data 字段说明**：

| 字段         | 类型   | 可空 | 说明 |
|--------------|--------|------|------|
| status       | string | 否   | 认证状态：`none`（未提交）、`pending`（审核中）、`approved`（已通过）、`rejected`（已驳回） |
| nextAction   | string | 否   | 下一步动作：`submit`（提交认证）、`view`（查看状态）、`resubmit`（重新提交）、`done`（已完成） |
| companyName  | string | 可空 | 企业名称（status 非 none 时返回） |
| creditCode   | string | 可空 | 统一社会信用代码（status 非 none 时返回） |
| contactPhone | string | 可空 | 联系电话（status 非 none 时返回） |
| createdAt    | string | 可空 | 提交时间（status 非 none 时返回） |
| auditTime    | string | 可空 | 审核时间（status 为 approved 时返回） |
| rejectReason | string | 可空 | 驳回原因（status 为 rejected 时返回） |

**状态与动作对应关系**：

| status    | nextAction | 说明 |
|-----------|------------|------|
| none      | submit     | 未提交认证，引导用户提交 |
| pending   | view       | 审核中，可查看状态但不可重复提交 |
| approved  | done       | 已通过，认证完成 |
| rejected  | resubmit   | 已驳回，可基于上次资料重新提交 |

**失败响应**：
- 未登录：`{ "code": 401, "message": "请先登录" }`

## 五、收藏接口

当前约定：**收藏为登录用户的服务端持久化关系数据，未登录用户不可收藏。** 小程序不再以设备本地收藏记录作为主数据源。

### 5.1 收藏列表

| 项     | 值 |
|--------|-----|
| Method | GET |
| Path   | `{{baseUrl}}/api/favorites` |

- **请求**：Query `page`、`pageSize`；需鉴权。
- **响应**：`{ "code": 200, "data": { "total": number, "list": FavoriteItem[] } }`
- **说明**：
  - 只返回当前登录用户自己的收藏。
  - 仅返回当前仍能解析到源数据的收藏项；若原公告或文章已消失，则收藏列表中不再显示该项。
  - `FavoriteItem` 为前端收藏页可直接渲染的统一结构，包含 `viewType`、`favoritesType`、`site`、`favorited` 等字段。

### 5.2 收藏 / 取消收藏

| 项     | 值 |
|--------|-----|
| Method | POST |
| Path   | `{{baseUrl}}/api/favorites/toggle` |

- **请求体**：`{ "targetId": "列表/详情 id", "targetType": "bid" | "info", "targetSite"?: "site1|site2" }`。  
  - `bid` 类型必须传 `targetSite`，用于唯一定位 `(site, id)` 公告。
  - `info` 类型主要用于文章收藏；若信息详情来自公告详情页，可附带 `targetSite` 供后端解析。
  - 若当前已收藏则取消收藏，未收藏则收藏。
- **响应**：`{ "code": 200, "data": { "favorited": true | false } }`。
- **失败响应**：
  - 未登录：HTTP 200，body 为 `{ "detail": { "code": 401, "message": "请先登录" } }`
  - 参数错误：`{ "code": 400, "message": "targetType 仅支持 bid 或 info" }`

### 5.3 收藏状态字段

以下接口在携带有效用户 token 时，都会返回当前用户视角下的 `favorited` 字段，作为页面心形状态的服务端真值来源：

- `GET /api/list`
- `GET /api/detail/bid/:id`
- `GET /api/detail/info/:id`
- `GET /api/articles`
- `GET /api/articles/:id`

## 六、基础数据接口（可选）

- **地区**：若筛选需要地区列表，可提供 `GET {{baseUrl}}/api/dict/regions`，返回树形或扁平结构（如 `{ "code": "510100", "name": "成都市" }`），与《原始数据接口文档》中 site2 getRegionJson 对应。
- **采购方式**：若「政府采购-采购公告」筛选需要采购方式选项，可提供 `GET {{baseUrl}}/api/dict/purchase-manner`，与 site2 getDictInfo 对应。

**否则**：在文档中注明「地区/采购方式等由后端在列表接口中内联或暂不提供」，前端筛选区仅展示后端实际支持的参数。

## 七、与原始数据接口的映射说明

- **列表**：前端 `category` + 筛选参数由后端映射为 site1-list 的 condition/time 或 site2 的 noticeType、operationStartTime/EndTime、regionCode、purchaseManner 等；响应字段与《原始数据接口文档》中 site1 result.records、site2 data.rows 落库字段对应（见 1.4 列表单条字段表）。
- **招投标详情**：由后端根据 id 或 site 从 site1 静态 JSON 或 site2 getInfoById/selectInfoByOpenTenderCode 取得数据并归一为 2.4 字段表；部分字段（如 projectName、enrollStart/enrollEnd）若原始数据未提供，需**待后端补充**或标注「仅展示已有字段」。
- **信息展示详情**：与 site2 采购意向公开详情 getInfoById 的 data 对应；originUrl 若 site2 未直接返回，可**待后端补充**拼装规则。
